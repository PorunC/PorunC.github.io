## 一、总体架构：四层记忆协同

Hermes Agent 的记忆系统不是单一的文件或数据库，而是四层记忆在协同工作：

| 层级 | 名称 | 载体 | 本质 |
|------|------|------|------|
| L1 | 声明性记忆 | MEMORY.md / USER.md | Agent"知道"什么事实 |
| L2 | 程序性记忆 | SKILL.md (Skill 系统) | Agent"会"什么方法论 |
| L3 | 会话记忆 | SQLite + FTS5 | Agent"做过"什么历史 |
| L4 | 训练记忆 | batch_runner + trajectory_compressor | Agent"如何变得"更好 |

前三层是运行时能力，第四层是进化能力。这四层组合起来，Agent 才真正具备了**从自己的行为中学习的闭环能力**。

这也是 Hermes 与 Claude Code 等竞品的根本差异：Claude Code 有文件级记忆但没有 RL 数据流水线，而 Hermes 把记忆、技能、训练串在了一条链路上。

## 二、MemoryManager——记忆中枢调度器

`agent/memory_manager.py` (640 行) 是整个记忆系统的枢纽。

### 2.1 架构约束

核心约束：

- **内置 provider 永远在第一位**。`add_provider()` 对 `"builtin"` 名称做特殊处理，永远接受。
- **外部 provider 最多一个**。尝试注册第二个会被拒绝并打 warning。

这个设计的现实考量是：如果有两个外部 provider 同时注册了同名 tool schema，模型调用 `memory` 工具时会路由给谁？更微妙的是——Honcho 的 `prefetch()` 和 Mem0 的 `prefetch()` 同时往 system prompt 注入记忆片段，LLM 看到两段可能冲突的上下文，该信谁？这些问题没有框架级的好解法，所以 Hermes 直接禁止了多外部 provider 共存。

注册同时建立了 **tool_name → provider** 的 O(1) 路由索引。当模型调用 memory tool 时，MemoryManager 直接查字典路由，不需要遍历 provider list。

几个关键洞察：

**`on_pre_compress` 是最精巧的钩子**。上下文压缩器准备丢弃旧消息时，MemoryManager 先让所有 provider 过一遍即将被丢弃的内容。返回的文本注入到压缩摘要的 prompt 里。这意味着：**压缩器压的是上下文窗口，但知识不会丢失**。

```python
# agent/memory_manager.py
def on_pre_compress(self, messages: List[Dict[str, Any]]) -> str:
    parts = []
    for provider in self._providers:
        try:
            result = provider.on_pre_compress(messages)
            if result and result.strip():
                parts.append(result)
        except Exception as e:
            logger.debug("Memory provider '%s' on_pre_compress failed: %s",
                         provider.name, e)
    return "\n\n".join(parts)
```

**`on_memory_write` 有循环防护**。广播内置记忆写入时，显式跳过 builtin provider 自身——因为 builtin 就是写入源头，再通知自己形成环路。**shutdown 是反序的**——后注册的 provider（通常是外部 provider）先关闭，因为外部 provider 可能依赖内置 provider 的数据状态做 flush。**每个钩子都有 try-except 包裹**——一个 provider 崩溃不会影响另一个。

### 2.3 Context Fencing 与 Streaming Context Scrubber

`agent/memory_manager.py` 中有一个不太显眼但极其关键的组件：**`StreamingContextScrubber`**。

当外部 provider 的 `prefetch()` 返回记忆内容时，这些内容被包裹在 `<memory-context>` 标签中注入到 model 的输入。但流式传输的场景下，一个 `<memory-context>` 可能在某个 chunk 中打开，在下一个 chunk 中关闭——正则表达式无法跨 chunk 匹配。

```python
# StreamingContextScrubber 是一个有限状态机
# 状态：_in_span (bool) — 当前是否在 <memory-context> 内部
# 缓存：_buf — 可能跨 chunk 截断的标签尾部
#
# feed(text): 返回可见部分，将可能截断的标签尾部缓存
# flush(): 流结束时，如果在 span 内则丢弃（安全优先），否则 flush 缓存
```

状态机逐 chunk 过滤，确保 `<memory-context>` 中的记忆上下文不会泄漏到用户界面上。这是流式场景下的安全防护，纯正则无法做到。

## 四、BuiltinMemoryProvider——Markdown 文件记忆的深度剖析

### 4.1 Frozen Snapshot 模式

`tools/memory_tool.py` (723 行) 中的 `MemoryStore` 实现了**冻结快照模式**——这个设计是 Hermes 记忆系统最重要的工程创新：

```python
class MemoryStore:
    def __init__(self, memory_char_limit=2200, user_char_limit=1375):
        self.memory_entries: List[str] = []
        self.user_entries: List[str] = []
        # 冻结快照——session 启动时捕捉，永不改变
        self._system_prompt_snapshot = {"memory": "", "user": ""}

    def load_from_disk(self):
        self.memory_entries = self._read_file(mem_dir / "MEMORY.md")
        self.user_entries = self._read_file(mem_dir / "USER.md")
        # 去重 + 安全扫描
        self.memory_entries = list(dict.fromkeys(self.memory_entries))
        sanitized_memory = self._sanitize_entries_for_snapshot(...)
        sanitized_user = self._sanitize_entries_for_snapshot(...)
        # 捕捉冻结快照
        self._system_prompt_snapshot = {
            "memory": self._render_block("memory", sanitized_memory),
            "user": self._render_block("user", sanitized_user),
        }
```

这个设计的关键洞察：

1. **system prompt 在 session 内永不改变**。即使 mid-session 写入 MEMORY.md，`format_for_system_prompt()` 返回的仍然是启动时的快照。这保证了 **prefix cache 不会在 session 中途失效**。

2. **双状态并行**：`_system_prompt_snapshot` (冻结) + `memory_entries` / `user_entries` (实时)。模型通过 `memory(action=read)` 看到实时状态，但 system prompt 中的记忆始终是启动时的快照。

3. **安全扫描仅在快照构建时执行一次**。任何命中威胁模式的 entry 被替换为 `[BLOCKED: ...]` 占位符，而原始文本保留在 `memory_entries` 中，用户可以通过 memory tool 看到被 block 的内容并手动删除。

### 4.2 双重安全防护

`load_from_disk()` 中的 `_sanitize_entries_for_snapshot` 使用 `tools/threat_patterns.py` (252 行) 中的共享威胁库以 `"strict"` scope 扫描每条 entry。这是两层防护：

- **写入时**：`_scan_memory_content()` 在 `add()` / `replace()` 时拦截
- **加载时**：`_sanitize_entries_for_snapshot()` 在 snapshot 构建时再次拦截

即使攻击者绕过了写入时扫描（比如通过 `patch` 工具直接修改文件），加载时扫描仍然会捕获。双层保证了 supply chain 攻击也无法通过文件直接注入 system prompt。

### 4.3 External Drift Detection

```python
def _detect_external_drift(self, target):
    # 信号 1: 往返不一致——re-parse 再序列化后 bytes 不同
    # 信号 2: 单条 entry 超限——外部写入者追加了大块内容

    parsed = [e.strip() for e in raw.split(ENTRY_DELIMITER) if e.strip()]
    roundtrip = ENTRY_DELIMITER.join(parsed)
    max_entry_len = max((len(e) for e in parsed), default=0)
    drift_detected = (raw.strip() != roundtrip) or (max_entry_len > char_limit)
```

这个机制解决了一个真实痛点（issue #26045）：**多个会话同时写入同一份 MEMORY.md 时**，一个会话的写入可能会覆盖另一个会话的内容。检测到 drift 时，将原始文件备份为 `.bak.<timestamp>`，然后拒绝写入并返回包含详细 remediation 的错误信息。这是一个比悲观锁更安全的策略：检测到不确定性时放弃写入，而不是强制覆盖。

### 4.4 原子写入

```python
@staticmethod
def _write_file(path, entries):
    content = ENTRY_DELIMITER.join(entries)
    fd, tmp_path = tempfile.mkstemp(dir=str(path.parent),
                                     suffix=".tmp", prefix=".mem_")
    with os.fdopen(fd, "w") as f:
        f.write(content); f.flush(); os.fsync(f.fileno())
    atomic_replace(tmp_path, path)
```

使用 `os.replace()` 原子替换文件。读者总是看到完全写入的内容（旧文件或新文件），永远不会看到半写内容。这是一个关键的正确性保证——记忆文件可能被多个进程共享。

### 4.5 § 分隔符的设计权衡

使用 `§` (section sign, U+00A7) 作为 entry 分隔符是一个有趣的设计取舍：

| 分隔符 | 优点 | 缺点 |
|--------|------|------|
| 空行 | 人类可读性好 | `\n\n` 可能在 entry 中出现 |
| 水平线 `---` | Markdown 原生 | YAML frontmatter 二义性 |
| UUID 标记 | 唯一性最好 | 人类不可读，输出膨胀 |
| **`§`** | 人类可读 + 极低概率出现在内容中 | 长度 2 字节 |

`§` 在实际使用中足够可靠——用户几乎不会在记忆内容中输入这个字符。而且在 `_detect_external_drift` 的信号 1 中，如果有误用也会被检测到。

## 五、SQLite + FTS5 持久化——SessionDB 深度分析

`hermes_state.py` (3449 行) 中的 `SessionDB` 是整个 Agent 的运行时数据库，承载会话状态、消息历史、token 统计、平台绑定等所有结构化数据。

### 5.1 WAL 模式与 NFS 回退

```python
def apply_wal_with_fallback(conn):
    # 先探测 on-disk mode
    try:
        current_mode = conn.execute("PRAGMA journal_mode").fetchone()
        if current_mode and current_mode[0] == "wal":
            return "wal"  # 其他进程已经设了 WAL
    except: pass

    try:
        conn.execute("PRAGMA journal_mode=WAL")
        return "wal"
    except sqlite3.OperationalError as exc:
        msg = str(exc).lower()
        if "locking protocol" in msg:
            conn.execute("PRAGMA journal_mode=DELETE")
            return "delete"
```

WAL 在 NFS 上不工作。`apply_wal_with_fallback()` 先探测已有 journal mode，再尝试设置，仅在确认是 `"locking protocol"` 时才回退。并且对同一个 db_label 只打一次 WARNING，避免 kanban_db 每操作一次就刷一遍 errors.log。

### 5.2 抖动重试打破护航效应

```python
class SessionDB:
    _WRITE_MAX_RETRIES = 15
    _WRITE_RETRY_MIN_S = 0.020   # 20ms
    _WRITE_RETRY_MAX_S = 0.150   # 150ms

    def _execute_write(self, fn):
        for attempt in range(self._WRITE_MAX_RETRIES):
            try:
                with self._lock:
                    self._conn.execute("BEGIN IMMEDIATE")
                    try:
                        result = fn(self._conn)
                        self._conn.commit()
                    except BaseException:
                        self._conn.rollback()
                        raise
                return result
            except sqlite3.OperationalError as exc:
                if "locked" in str(exc).lower():
                    jitter = random.uniform(0.020, 0.150)
                    time.sleep(jitter)
                    continue
                raise
```

这个机制解决了 **convoy effect** (护航效应)：当多个 writer 使用 SQLite 内置的 busy handler 时，大家都按固定间隔重试，每次都在同一时刻撞车。随机抖动 (20-150ms) 让竞争者的重试时间自然错开，显著提高高并发下的吞吐量。

`BEGIN IMMEDIATE` 在事务开始时就抢写锁，而不是等到 commit 时才发现冲突。这缩短了锁持有时间：冲突越早暴露，重试越快。

### 5.3 被动 WAL Checkpoint

```python
def _try_wal_checkpoint(self):
    """PASSIVE checkpoint，不阻塞，不抛异常"""
    with self._lock:
        self._conn.execute("PRAGMA wal_checkpoint(PASSIVE)")
```

每 50 次写入做一次 PASSIVE checkpoint。只写回那些没有其他连接在读的 WAL 帧，有被占用的就跳过。

### 5.4 双重 FTS5 索引

```sql
-- 表 1: 标准 FTS5 (unicode61 tokenizer)
CREATE VIRTUAL TABLE IF NOT EXISTS messages_fts USING fts5(content);

-- 表 2: Trigram FTS5 (支持 CJK 子串搜索)
CREATE VIRTUAL TABLE IF NOT EXISTS messages_fts_trigram USING fts5(
    content, tokenize='trigram'
);
```

标准 FTS5 的 `unicode61` tokenizer 会把 CJK 字符拆成单个 token，导致中文短语搜索失效。Trigram tokenizer 创建 3 字符重叠序列，任何子串都能匹配。两个索引通过 trigger 自动同步。

搜过时优先用 trigram 表，fallback 到标准表。搜索查询还经过消毒处理——保护引号短语、剥离不匹配的 FTS5 特殊字符、用引号包裹连字符/点号词组保持短语语义。

### 5.5 Schema 迁移

版本控制用 `schema_version` 表实现（当前版本 14）。迁移均为 `ALTER TABLE ADD COLUMN` 式：

```python
if current_version < 6:
    for col_name, col_type in [
        ("reasoning", "TEXT"),
        ("reasoning_details", "TEXT"),
        ("codex_reasoning_items", "TEXT"),
    ]:
        cursor.execute(f'ALTER TABLE messages ADD COLUMN "{col_name}" {col_type}')
```

v6 加的 `reasoning` 字段将 assistant 的推理链持久化到数据库，使得 gateway 会话 reload 时 reasoning 不会丢失。对于 OpenRouter、OpenAI 的 thinking 模式，这是多轮推理连续性的关键。

```python
        hint = " (state.db may be on NFS/SMB/FUSE)"
    return f"{prefix}: {cause}{hint}."
```

`SessionDB.__init__` 失败时记录错误原因（只 SET 不 CLEAR），`/resume`、`/title`、`/history` 等 slash command 在 `self._session_db is None` 时用 `format_session_db_unavailable()` 生成带原因的友好错误信息。这让用户知道"锁定协议错误"意味着 state.db 可能在 NFS 上，而不是看到一个裸的"Session database not available"。

## 六、BatchRunner——RL 训练数据工厂

`batch_runner.py` (1321 行) 负责从 prompt 数据集批量生成 Agent 行为轨迹。

### 6.1 工具集采样

```python
selected_toolsets = sample_toolsets_from_distribution(config["distribution"])
agent = AIAgent(
    enabled_toolsets=selected_toolsets,
    skip_context_files=True,   # 不让 SOUL.md/AGENTS.md 污染轨迹
    skip_memory=True,          # 批量跑不用持久记忆
)
```

每个 prompt 不是从固定工具集跑的，而是从分布中采样。`skip_memory=True` 是关键：批量生成训练数据时，持久记忆会干扰每个独立样本。每个 prompt 应该是一个干净的独立试验。

### 6.2 推理覆盖过滤

```python
reasoning = result.get("reasoning_stats", {})
if not reasoning.get("has_any_reasoning", True):
    discarded_no_reasoning += 1
    continue
```

如果一条轨迹的所有 assistant turn 都没有推理过程（既没有 `<REASONING_SCRATCHPAD>` 也没有 native thinking token），直接丢弃。**没有推理的轨迹对 RL 训练没有价值**，留着只会引入噪声。

推理统计检测两种推理形式：
- `<REASONING_SCRATCHPAD>` 在 content 中（Hermes 的自定义推理标记）
- `msg["reasoning"]` 非空（OpenAI / Anthropic 的原生 thinking token）

### 6.3 断点续跑与内容匹配

```python
def _scan_completed_prompts_by_content(self):
```

## 七、TrajectoryCompressor——RL 训练数据压缩

`trajectory_compressor.py` (1508 行) 负责在保留训练信号的前提下将轨迹压短。

### 7.1 分区保护策略

```python
@dataclass
class CompressionConfig:
    target_max_tokens: int = 15250
    summary_target_tokens: int = 750
    protect_first_system: bool = True
    protect_first_human: bool = True
    protect_first_gpt: bool = True
    protect_first_tool: bool = True
    protect_last_n_turns: int = 4
```

```
原始轨迹：
[system] [human] [gpt] [tool] [gpt] [tool] ... [gpt] [tool] [gpt] [tool]
 ↑ 保护 ↑  ↑保护↑ ↑保护↑ ↑保护↑                      ↑── 保护最后4轮 ──↑
                               ↑────── 可压缩区域 ──────↑

压缩后：
[system] [human] [gpt] [tool] [CONTEXT SUMMARY] ... [gpt] [tool] [gpt] [tool]
                                   ↑ 一条摘要消息替换 N 条被压缩的 turn
```

| 保护项 | 理由 |
|--------|------|
| 首条 system | 人格定义和工具 schema |
| 首条 human | 任务描述 |
| 首条 gpt | 模型对任务的初始理解 |
| 首条 tool | 第一次工具探索的结果 |
| 最后 4 轮 | 最终行动和结论（RL reward signal） |

### 7.2 压缩量刚好够用

```python
tokens_to_save = total_tokens - self.config.target_max_tokens
target_tokens_to_compress = tokens_to_save + self.config.summary_target_tokens

for i in range(compress_start, compress_end):
    accumulated_tokens += turn_tokens[i]
    compress_until = i + 1
    if accumulated_tokens >= target_tokens_to_compress:
        break  # 够了就停，不多压
```

压缩只在中间区域动刀，而且只压到刚好够用的量。这是一个最小化信息损失的策略——不必要的压缩只会丢失训练信号。

### 7.3 LLM 摘要生成

```python
summary = await self._generate_summary_async(content_to_summarize, metrics)
compressed.append({
    "from": "human",
    "value": summary  # "[CONTEXT SUMMARY]: ..."
})
```

被压缩的 turn 不是直接丢弃，而是用 LLM 生成摘要替换。摘要 prompt 要求 LLM 以中立视角描述 assistant 做了什么、获得了什么关键信息、做了什么决策。目标约 750 tokens，以 factuality 为主。

### 7.4 异步并行处理

```python
semaphore = asyncio.Semaphore(self.config.max_concurrent_requests)  # 默认 50

async def process_single(file_path, entry_idx, entry, ...):
    async with semaphore:
        processed_entry = await asyncio.wait_for(
            self.process_entry_async(entry),
            timeout=self.config.per_trajectory_timeout  # 300s
        )
```

50 个并发 API 调用同时跑摘要生成。每条轨迹有独立的 5 分钟超时，超时的轨迹直接跳过不写入输出——比让一条卡住的轨迹拖住整个流水线好得多。

## 九、对比分析

### 9.1 Hermes Agent vs Claude Code

| 维度 | Hermes Agent | Claude Code |
|------|-------------|-------------|
| **记忆存储** | MEMORY.md + USER.md + 8 种外部 provider | CLAUDE.md + MEMORY.md |
| **记忆架构** | 插件化，1 内置 + 1 外部（最多） | 固定双文件，无插件 |
| **记忆召回** | 内置全量注入 / 外部 query 召回 | 全量注入 system prompt |
| **持久化** | SQLite WAL + FTS5 全文搜索（14 版本 schema） | 纯文件系统 |
| **Skill 系统** | Agent 自主创建 + 安全扫描 + 分类目录 | 无程序性记忆 |
| **RL 数据采集** | batch_runner + trajectory_compressor | 无公共 RL pipeline |
| **上下文压缩** | `on_pre_compress` 钩子提取洞察 | 压缩时保留 CLAUDE.md |
| **多后端** | 8 种 (Honcho / Mem0 / Holographic 等) | 无外部记忆后端 |
| **安全防护** | 70+ 正则 + 信任矩阵 + LLM 审计 | 依赖 Anthropic 自身安全层 |

核心差异：Claude Code 的记忆是**声明式的**（告诉 Agent 你是谁、规则是什么），Hermes 的 Skill 是**过程式的**（告诉 Agent 遇到这类问题该怎么一步步做）。Claude Code 没有主动记忆能力——它的记忆完全依赖人类手动编辑，模型本身不会自动提取和存储有用信息。

还有一个动态召回的差异。Claude Code 把 CLAUDE.md 的全部内容塞进每次对话的 system prompt，文件大了就浪费 token。Hermes 的外部 provider 支持 `prefetch()` 语义召回，只拉跟当前 query 相关的记忆片段。

### 9.2 Hermes Agent vs OpenClaw QMD 索引

| 维度 | Hermes Agent Memory | OpenClaw QMD |
|------|--------------------|--------------|
| **领域** | Agent 经验积累 | 知识库问答 |
| **索引粒度** | 会话 turn 级别 | QMD 文档级别 |
| **检索** | FTS5 + provider 向量检索 | 关键词 + 向量 |
| **写入时机** | 每轮对话后异步 sync | 文档变更时批量重建 |
| **记忆类型** | 事实记忆 + 程序性记忆 (Skill) | 知识库检索，无程序性记忆 |
| **更新节奏** | 实时 (每轮对话) | 离线批量 |

OpenClaw 解决的是"如何让 Agent 获取外部知识"，Hermes 解决的是"如何让 Agent 积累自身经验"。前者是 RAG，后者更接近 episodic memory + procedural memory。QMD 索引是面向知识库问答设计的，最小单元是一个 QA 对；Hermes 的记忆系统面向 Agent 工作流，最小单元是一次对话中提取的事实或过程。

### 关键总结

1. **插件化不是肤浅的抽象**。MemoryManager 的 `add_provider` 强制"最多一个外部 provider"的原因是：多外部 provider 的 tool schema 冲突和 prefetch 上下文打架的问题没有框架级的好解法。这是一个诚实架构约束。

2. **Frozen snapshot 是 cache-friendly 的精华设计**。内置记忆使用冻结快照而非实时注入，这是 system prompt prefix caching 能生效的关键。一个 session 内 system prompt 不变化，LLM provider 的 KV cache 就不会在每轮对话时失效。

3. **SQLite 的抖动重试是一个优雅的工程 hack**。`BEGIN IMMEDIATE` + 20-150ms 随机退避的组合，比 SQLite 内置的 busy handler 在高并发下吞吐量更高，因为随机抖动的自然错开效果优于确定性调度。

4. **RL pipeline 的过滤比压缩更关键**。在 batch_runner 中，推理覆盖过滤和幻觉工具过滤直接去除了对训练有害的噪声数据，这些"负过滤"对训练质量的影响可能比轨迹压缩更大。

5. **Skill 安全扫描是双层防御**。70+ 正则捕获已知模式 + LLM 审计捕获上下文相关的新型攻击，且 LLM 只能提升不能降低严重度。这是 defense-in-depth 在 Agent 系统中的应用。

6. **整条链路的工程取舍**。从 LLM 调用看，memory tool 的 schema 描述用了大段的"when to save"指南（约 300 字符），这是一种 prompt engineering 的投资——没有这层引导，模型可能把任务进度写到 MEMORY.md 里，污染记忆系统。从 DB 层面看，双重 FTS5 索引增加了写放大（insert trigger 要更新三个表），但换来了中英文搜索的兼得。这些取舍的共通逻辑是：**把约束和引导放在离数据源头最近的环节**，后面才能跑得稳。
