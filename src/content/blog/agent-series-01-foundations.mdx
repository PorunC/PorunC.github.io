Agent 最容易被误解的地方，是把它看成“会调用工具的聊天机器人”。这个说法不算错，但太浅。真正的 Agent 不是多了一个搜索工具，也不是在 prompt 里写一句“请一步步思考”，而是把模型放进一个目标、环境、记忆、工具和反馈组成的闭环里，让它能在运行时决定下一步怎么做。

`01_基础知识` 这个模块的价值，就在于把 Agent 的边界画清楚：它和传统 AI、LLM 应用、Chatbot、工作流、多 Agent 系统分别有什么区别；它为什么需要记忆、工具、环境交互和多步推理；它的能力上限和失败边界又在哪里。

## 1. Agent 的本质：目标驱动的闭环系统

传统 AI 往往是输入到输出的映射：给图片，输出分类；给文本，输出标签；给问题，输出答案。LLM 应用则通常是一次或多次对话：用户输入一段话，模型生成一段文本。Agent 的关键变化是：系统不只生成文本，还要围绕目标持续行动。

所以判断一个系统是不是 Agent，可以看四个问题：

| 问题 | 如果答案是否定的，它更像什么 |
| --- | --- |
| 它是否有明确目标，而不只是被动回答 | Chatbot |
| 它是否能感知上下文、工具结果或环境状态 | 普通 LLM 应用 |
| 它是否能在运行时选择下一步行动 | 固定 workflow |
| 它是否能根据反馈调整计划 | 脚本或自动化流程 |

Agent 的核心不是“智能感”本身，而是闭环：目标输入，系统感知，模型决策，工具执行，环境返回观察，系统更新记忆，再继续决策。只要这个循环存在，Agent 就开始从“说”变成“做”。

## 2. LLM 与 Agent：大脑和系统的区别

LLM 是 Agent 的推理引擎，但不是 Agent 的全部。模型负责理解目标、分析上下文、生成计划、选择工具、解释结果。Agent 系统还要负责更多工程问题：工具注册、权限控制、状态保存、执行调度、错误恢复、终止判断、日志追踪。

这也是为什么同一个模型，在不同 Agent Harness 或框架里表现差异很大。模型能力决定上限，系统设计决定下限。一个强模型如果没有好的工具描述、没有状态管理、没有失败处理，也会在多步任务中反复绕路；一个中等模型如果被放进清晰的任务边界、可靠的工具和严格的反馈循环里，反而能稳定完成窄领域任务。

可以把两者关系理解成：

| 层级 | 负责什么 | 典型问题 |
| --- | --- | --- |
| LLM | 语言理解、推理、决策生成 | 下一步该做什么 |
| Agent | 目标、工具、状态、环境、执行循环 | 怎么把下一步真正执行好 |
| 产品系统 | 用户、权限、成本、体验、合规 | 怎么让它可用、可靠、可维护 |

## 3. Agent 的核心组件

基础模块里反复强调五个核心组件：推理引擎、工具集、记忆系统、规划模块、执行控制。再从完整架构视角看，可以扩展成感知、规划、决策、执行、记忆、反馈六个模块。

这两种说法并不冲突。五组件更适合解释 Agent 的组成，六模块更适合解释 Agent 的运行过程。

| 组件 | 作用 | 工程关键点 |
| --- | --- | --- |
| 推理引擎 | 理解目标、选择行动、反思结果 | 模型选择、上下文拼装、温度控制 |
| 工具集 | 让 Agent 访问外部世界 | Schema、权限、幂等、结果格式 |
| 记忆系统 | 保持任务连续性与跨会话经验 | 短期上下文、工作状态、长期检索 |
| 规划模块 | 把复杂目标拆成可执行步骤 | 粒度、依赖、重规划 |
| 执行控制 | 让循环按规则运行和停止 | max steps、错误恢复、终止判断 |

很多“伪 Agent”缺的不是工具，而是执行控制。模型会调用搜索不代表它能完成任务。真正困难的是：什么时候搜索，搜不到怎么办，结果互相矛盾怎么办，已经花太多 token 是否要停止，最终答案是否满足原始目标。

## 4. 自主性不是无限自由，而是受控决策

“自主 Agent”容易让人想到完全放手，让模型自己随便跑。生产系统里恰好相反：Agent 的自主性越强，越需要边界。它可以自主选择路径，但不能自主突破权限；可以自主重试，但不能无限消耗预算；可以自主调用工具，但不能绕过审计。

目标驱动型 Agent 的关键是把“完成什么”交给目标，把“怎么做”留给运行时决策，同时用系统边界约束行动空间。

一个健康的目标驱动 Agent 通常具备这些约束：

| 约束 | 作用 |
| --- | --- |
| 工具白名单 | 只暴露任务需要的能力 |
| 资源预算 | 控制 token、时间、成本、步数 |
| 状态机 | 限定合法阶段和转移 |
| 验证器 | 判断结果是否真的完成 |
| 人类审批 | 保护高风险操作 |

自主性不是“模型想干什么就干什么”，而是“在可审计的行动空间内，让模型决定最合适的下一步”。

## 5. 记忆、环境与多步推理

Agent 之所以需要记忆，是因为多步任务会产生中间状态。没有记忆，模型每一步都像第一次见到任务，只能依赖当前 prompt 里剩下的内容。

可以把记忆分成三层：

| 记忆类型 | 保存什么 | 典型载体 |
| --- | --- | --- |
| 短期记忆 | 当前对话和最近步骤 | Context window |
| 工作记忆 | 当前任务的计划、变量、工具结果 | State store、Redis、内存对象 |
| 长期记忆 | 用户偏好、历史经验、知识片段 | 向量库、数据库、文件 |

环境交互则让 Agent 摆脱模型参数的限制。模型内部知识可能过期，计算可能不精确，对外部状态没有直接感知。工具调用、API 查询、文件读取、数据库检索、代码执行等操作，就是 Agent 的“感知与行动”通道。

多步推理不是让模型在脑内想更久，而是把任务拆成可验证的步骤：每一步都产生观察，每一步都可以被纠错。

## 6. 单 Agent 与多 Agent

单 Agent 是基础单元，多 Agent 是组织方式。不要因为“多 Agent”听起来高级就过早拆分。多 Agent 的价值在于角色分工和并行协作，但代价是通信、协调、冲突解决和状态一致性。

适合多 Agent 的场景通常有几个特征：任务能自然拆成多个专业子领域；子任务之间接口清楚；各角色需要不同工具或不同提示策略；并行带来的收益大于协调成本。

如果一个任务本质上只是线性多步执行，单 Agent 加清晰规划往往更稳。如果任务像报告生成、代码评审、调研分析、测试验证这种存在角色分工，多 Agent 才有空间。

## 7. Agent 的能力边界

Agent 不是魔法。它最适合的问题有三个特点：

| 条件 | 说明 |
| --- | --- |
| 目标可描述 | 用户能说清楚想要什么 |
| 环境可观察 | Agent 能通过上下文、工具或检索获得必要信息 |
| 结果可验证 | 系统或人能判断是否完成 |

反过来，Agent 容易失败的任务也很清楚：目标含糊、缺少关键数据、工具权限不足、没有反馈信号、评价标准主观且无法校验、执行动作风险过高。

所以基础知识最后要落到一个工程判断：不要问“Agent 能不能做这件事”，而要问“目标、信息、工具、反馈、验证是否都在系统里”。如果这五件事不齐，模型再强也只是更会猜。

## 8. 学习这个模块的正确姿势

学习 Agent 基础，不要先背框架名。先把范式想清楚：

1. Agent 是闭环系统，不是一次性文本生成。
2. LLM 是推理引擎，不是完整应用。
3. 工具让 Agent 能行动，记忆让 Agent 能连续，反馈让 Agent 能修正。
4. 自主性必须和权限、预算、审计一起设计。
5. 多 Agent 是组织复杂任务的方法，不是默认选项。

真正理解这几条之后，再看 RAG、工具调用、框架、系统设计、安全和 Harness，才会发现它们不是零散知识，而是在给同一个闭环补不同的工程能力。

## 9. 外部资料补强：从“组件清单”到“系统范式”

Lilian Weng 在《LLM Powered Autonomous Agents》中把 LLM Agent 概括为 LLM 加 Planning、Memory、Tool use 的组合，这和本模块的五组件模型高度一致。差别在于，工程落地时还要再补一层执行控制：预算、终止、权限、日志和恢复。如果只谈 Planning、Memory、Tool use，很容易得到一个能演示的 Agent；如果再加执行控制，才开始接近生产系统。

Anthropic 的《Building Effective AI Agents》对“workflow”和“agent”做了非常有用的区分：workflow 是预定义路径，agent 是由模型根据环境反馈动态决定路径。这个区分能帮助我们避免两个极端：把所有 LLM 应用都叫 Agent，或者一上来就追求完全自主。真实项目里，最好先用 workflow 承载稳定路径，再只在需要动态判断的节点里引入 Agent。

ReAct 论文进一步说明了 Agent 的基础循环为什么有效：推理轨迹帮助模型跟踪计划和处理异常，行动让模型能访问外部信息和环境反馈。因此，Agent 的“智能”不是只发生在模型内部，而是发生在“推理加行动加观察”的交替过程中。基础知识模块里的环境交互、工具使用、多步推理，其实都可以回到这条循环理解。

Generative Agents 论文则展示了更长生命周期 Agent 的另一面：记忆、反思和计划会产生更像“连续行为”的系统效果。对工程系统来说，这个启发不是要模拟人类生活，而是说明长期记忆如果没有反思和检索策略，就只是日志；只有能被总结、召回、用于计划，记忆才真正参与 Agent 行为。

## 10. 深度补充：为什么 Agent 会从聊天应用里长出来

如果只看技术栈，Agent 很像“LLM + 工具 + 记忆”。但从应用演进看，它其实是在解决 ChatGPT 破圈后立刻暴露出来的三个产品问题。

第一个问题是“用户要的不是答案，而是完成”。用户让模型“帮我整理竞品报告”“帮我修这个 bug”“帮我安排一次会议”，真正需求不是得到一段解释，而是让一系列动作被执行。普通 Chatbot 可以告诉你怎么做，但每一步仍要人复制、粘贴、查资料、跑命令。Agent 的出现，就是把这些手动中间步骤纳入系统循环。

第二个问题是“对话没有状态边界”。聊天窗口里历史越堆越多，模型既不知道哪些信息仍然有效，也不知道哪些约束只属于上一个任务。Agent 把对话重新组织成任务：目标、状态、工具、记忆、验证、停止条件。这样系统才能知道“我正在做哪个任务”“做到哪一步”“还差什么证据”。

第三个问题是“模型知识不能直接变成环境事实”。模型可以说某个命令可能有用，但它不知道当前仓库测试是否通过；可以说某个订单可能已发货，但它没有查订单系统。Agent 用工具把模型判断接到外部环境，形成观察和行动的闭环。

真实应用里，Agent 通常不是一个万能人格，而是几类窄任务系统：

| 应用方式 | 它解决的问题 | 为什么要用 Agent |
| --- | --- | --- |
| Coding Agent | 修 bug、补测试、解释仓库、改代码 | 需要读文件、跑命令、根据失败继续修 |
| Research Agent | 搜索、筛选来源、整理证据、生成报告 | 信息不在模型内部，需要多轮检索和引用 |
| Customer Support Agent | 查订单、查政策、创建工单、转人工 | 需要在对话中调用业务系统 |
| Data Analyst Agent | 写 SQL、查数、画图、解释异常 | 需要工具执行和结果验证 |
| Workflow Agent | 处理审批、通知、表单和状态流转 | 需要跨系统行动和权限控制 |

所以，Agent 基础知识的核心不是“模型能不能自主思考”，而是“系统有没有把目标完成过程工程化”。一个 Agent 应该至少有三层边界：

1. 任务边界：它被授权完成什么，不负责什么。
2. 行动边界：它能用哪些工具，能改哪些数据。
3. 验证边界：它如何证明自己完成，而不是自己宣布完成。

这也解释了为什么 AutoGPT/BabyAGI 当年出圈后，社区很快从“完全自主”回到“受控自主”。没有边界的自主只会烧 token；有边界、有工具、有验证的自主，才是可落地的 Agent。

## 11. 面试追问视角：可以怎么回答

如果被问“Agent 到底解决了 Chatbot 的什么问题”，可以答：

> Chatbot 主要解决自然语言交互，Agent 解决的是多步任务完成。它把目标、工具、状态、反馈和终止条件组织成闭环，让模型不只是回答，而是能在环境里行动并根据结果继续调整。

如果被问“为什么不是普通 workflow”，可以答：

> 普通 workflow 的路径由开发者预先写死，Agent 的下一步由模型根据上下文和观察结果动态决定。生产系统里两者通常混合：外层 workflow 控制边界，内部 Agent 节点处理需要语义判断的部分。

如果被问“Agent 的最小可用架构是什么”，可以答：

> 最小架构是 LLM 决策、工具执行、状态记录和终止判断。没有工具只是聊天，没有状态不能多步，没有终止判断会无限循环。记忆、规划、多 Agent 是在任务复杂后再加的能力。
