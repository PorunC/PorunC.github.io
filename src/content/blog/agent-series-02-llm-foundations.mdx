Agent 系统的上限，很大一部分来自 LLM。模型理解目标的能力、跨步骤推理的能力、遵循工具 Schema 的能力、处理长上下文的能力，都会直接影响 Agent 的任务成功率。

但 LLM 基础模块并不是为了让我们背 Transformer 细节，而是为了回答一个工程问题：当我们把模型放进 Agent 循环里，它的概率生成机制会带来哪些优势和风险。

## 1. LLM 的能力来自概率建模

大语言模型的基本训练目标是预测下一个 token。它不是数据库，不是规则引擎，也不是严格意义上的逻辑证明器。它学习的是海量文本中的模式：语义关系、推理习惯、代码结构、问答格式、工具调用样式。

这带来两个同时存在的事实：

| 事实 | 对 Agent 的影响 |
| --- | --- |
| 模型擅长从上下文中归纳模式 | 能理解自然语言目标并生成计划 |
| 模型输出是概率性的 | 同一任务可能走出不同路径，需要评估和控制 |

Agent 的工程设计，很多时候就是在利用前者、约束后者。我们希望模型能灵活理解任务，又不希望它随意编造工具参数、忽略约束或输出不可验证的结论。

## 2. Token 与上下文窗口决定信息预算

Token 是 LLM 的基本输入输出单位。用户消息、系统指令、工具描述、历史轨迹、RAG 检索内容、工具返回结果，最终都要挤进上下文窗口。

Agent 比普通聊天更容易消耗上下文，因为它不仅有对话，还会产生执行轨迹：

| 上下文内容 | 为什么会变长 |
| --- | --- |
| System Prompt | 包含角色、规则、输出格式、安全策略 |
| Tool Schema | 工具越多，描述和参数越占 token |
| ReAct 轨迹 | 每一步 Thought、Action、Observation 都会追加 |
| RAG Context | 检索片段需要带来源和证据 |
| Memory | 历史偏好和任务状态按需注入 |

所以 Agent 设计必须把上下文当预算管理。能用工具结果摘要的，不要塞原始大文本；能用检索按需注入的，不要把全知识库放进 prompt；能把工具分组的，不要一次暴露几十个工具。

## 3. Transformer 与 Attention 的工程含义

Transformer 的核心是 Attention。对 Agent 开发来说，重要的不是公式，而是它意味着模型会在上下文中分配注意力。信息越多，不代表效果越好。无关规则、重复日志、噪声检索片段都会稀释注意力。

这解释了几个常见现象：

| 现象 | 原因 |
| --- | --- |
| Prompt 很长但效果变差 | 有效信息密度下降 |
| 工具太多导致误调用 | 工具描述之间互相干扰 |
| RAG 检索太宽反而幻觉 | 无关证据进入上下文 |
| 早期关键约束被忽略 | 长上下文中的注意力衰减 |

优秀的 Agent 系统不是把信息都给模型，而是每一轮只给它“这一步需要的信息”。

## 4. 采样参数决定稳定性与探索性

Temperature、Top-K、Top-P 影响生成的随机性。对普通创作任务，高一点的随机性可能带来多样性；对 Agent 工具调用和生产任务，稳定性往往更重要。

可以按任务类型粗略分配：

| 任务 | 推荐策略 |
| --- | --- |
| 工具调用、结构化输出 | 低 temperature，强调格式稳定 |
| 分类、路由、规则判断 | 低 temperature，避免漂移 |
| 头脑风暴、文案创作 | 中等 temperature，保留变化 |
| 多方案规划 | 可以采样多次后 rerank |

Agent 的特殊之处在于，一次不稳定输出可能引发后续连锁错误。工具名选错、参数格式错、计划第一步跑偏，后面每一步都在错误轨道上。低随机性不是保守，而是降低控制流噪声。

## 5. 幻觉在 Agent 中更危险

LLM 幻觉通常指模型生成看似合理但事实错误的内容。在 Agent 场景里，幻觉不只体现在最终答案，还会出现在中间决策：

| 幻觉位置 | 例子 |
| --- | --- |
| 事实幻觉 | 编造不存在的政策、接口、函数 |
| 工具幻觉 | 调用未注册工具或虚构参数 |
| 计划幻觉 | 假设已经完成某一步，但其实没有 |
| 观察幻觉 | 忽略工具返回错误，继续当作成功 |
| 来源幻觉 | 引用 RAG 中不存在的证据 |

降低幻觉不能只靠 prompt 里写“不要幻觉”。更可靠的方法是把事实交给外部系统：实时信息通过搜索或数据库，私有知识通过 RAG，数学通过计算工具，代码通过测试，最终答案通过 verifier。

## 6. 指令微调与 RLHF 改变的是行为风格

预训练让模型学会语言和知识模式，指令微调让模型学会按用户要求回答，RLHF 让模型更符合人类偏好。对 Agent 来说，这些阶段影响的是模型是否愿意遵循格式、是否能拒绝危险请求、是否能稳定执行多轮任务。

但这也带来一个副作用：模型有时会“过度迎合”。它可能在信息不足时给出看似完整的答案，而不是主动要求检索；可能为了显得有帮助而跳过验证；可能在工具报错后编一个自然语言解释。

因此，Agent 系统需要把“诚实失败”写进控制逻辑：信息不足就继续检索，工具失败就返回 observation，无法完成就说明原因。不能把所有可靠性寄托在模型的礼貌和自觉上。

## 7. 模型选型不是排行榜问题

为项目选 LLM，不是看哪个模型综合榜单最高，而是看任务约束。一个生产 Agent 常常需要多模型协作：强模型做规划，小模型做分类，长上下文模型做文档分析，低延迟模型做实时回复，专门推理模型处理复杂规划。

选型维度可以这样拆：

| 维度 | 需要问的问题 |
| --- | --- |
| 任务类型 | 是聊天、代码、RAG、规划、摘要还是多模态 |
| 推理强度 | 是否需要长链路、多步骤、数学或逻辑推理 |
| 上下文 | 是否需要处理长文档、全仓库或长会话 |
| 延迟 | 用户能等 1 秒、3 秒还是 30 秒 |
| 成本 | 请求量上来后 token 成本是否可承受 |
| 隐私 | 数据是否允许进入第三方 API |
| 中文能力 | 主语言是否以中文为主 |

最稳妥的流程是：先用硬约束排除，再用业务评估集比较。通用 benchmark 只能给方向，真正决定上线的是自己的任务集合。

## 8. Agent 中的模型分工

复杂 Agent 不一定所有步骤都用同一个模型。模型路由是成本优化的重要手段。

| 角色 | 适合模型 |
| --- | --- |
| Planner | 强推理模型，重视全局计划 |
| Executor | 稳定、便宜、工具调用能力好的模型 |
| Router | 小模型或规则分类器 |
| Summarizer | 低成本长上下文模型 |
| Verifier | 判断严谨、保守的模型或规则系统 |
| RAG Answerer | 长上下文和事实遵循能力强的模型 |

这种分工的核心思想是：不要让最贵的模型做所有事。简单步骤用便宜模型，关键决策用强模型，最后用验证器兜底。

## 9. 这个模块的工程结论

LLM 基础知识最终要落到几个工程习惯：

1. 把模型输出当概率结果，不当确定事实。
2. 把上下文窗口当稀缺资源，不随意堆信息。
3. 工具调用和结构化输出尽量低随机性。
4. 用外部工具和验证器补足模型的事实与执行能力。
5. 用业务评估集选模型，而不是只看排行榜。
6. 建立模型路由，让强模型用在真正值钱的步骤上。

Agent 的上限由模型能力决定，但稳定性来自系统控制。理解 LLM 的概率本质，才能设计出不被模型随机性牵着走的 Agent。

## 10. 外部资料补强：模型机制如何落到 Agent 设计

Transformer 论文《Attention Is All You Need》提出的关键变化，是用注意力机制替代传统循环结构来建模序列关系。对 Agent 工程师来说，最直接的启发是：上下文组织会影响模型注意力分配。把工具、记忆、RAG 证据、历史轨迹全部堆进去，不等于模型会自动抓住重点。上下文工程本质上是在帮注意力机制“减负”。

InstructGPT/RLHF 论文说明了另一个重要事实：模型越大不等于越会按人类意图办事。指令微调和人类反馈能显著改善 helpful、honest、harmless 等行为，但模型仍会犯错。放到 Agent 场景中，这意味着我们既要利用模型的指令跟随能力，也不能把可靠性完全交给“模型已经对齐”。工具校验、外部 verifier、权限系统仍然不可省。

CoT 和 Self-Consistency 的研究说明，大模型在复杂推理上受“推理路径”影响很大。Agent 中可以借鉴这个思想：关键计划可以先生成多个候选，再用成本、风险、依赖和可验证性排序；关键判断也可以用多样化采样或独立 verifier 减少偶然错误。不过生产系统不应无限采样，因为每多一条推理路径都在增加延迟和成本。

因此，LLM 基础不是一组模型史知识，而是三条工程原则：上下文要有信息密度，模型输出要被当成概率结果，关键行为要用外部系统约束和验证。

## 11. 深度补充：LLM 在 Agent 里不是一个角色，而是一组能力供应商

很多 Agent 设计失败，是因为把 LLM 当成一个万能脑袋。真实系统里，LLM 更像一组可组合能力：理解、改写、分类、规划、抽取、生成、评估、总结。不同能力对模型的要求不同，也应该被放在不同位置。

| Agent 位置 | LLM 任务 | 对模型的要求 |
| --- | --- | --- |
| Router | 判断任务类型、选择流程 | 低延迟、稳定分类 |
| Planner | 拆目标、排依赖、识别风险 | 强推理、长上下文 |
| Executor | 生成工具调用和参数 | 遵循 Schema、低随机 |
| Extractor | 从文档/日志抽取结构化事实 | 格式稳定、召回细节 |
| Summarizer | 压缩上下文和工具结果 | 保真、会保留证据 |
| Verifier | 判断结果是否满足标准 | 保守、严格、少迎合 |
| Writer | 组织最终答案或报告 | 表达能力、风格控制 |

这就是为什么生产 Agent 经常采用多模型路由，而不是一个模型跑到底。Planner 可以用强模型，Router 和格式转换可以用便宜模型，Verifier 可以用另一个独立模型或规则系统。模型分工的目的不是炫技，而是把成本、延迟和质量拆开优化。

LLM 基础知识还解释了一个常见矛盾：为什么同一个 Agent 在 demo 里很惊艳，上线后却不稳定。原因在于 demo 通常只展示一次成功路径，而生产系统面对的是分布：不同输入、不同上下文、不同工具状态、不同模型采样结果。模型的概率性会被多步循环放大。

所以，Agent 中使用 LLM 的工程原则是：

1. 对结构化动作低温生成，对创意任务保留随机性。
2. 对高风险判断使用独立 verifier，不让同一模型自评。
3. 对长上下文任务做信息裁剪，而不是盲目堆材料。
4. 对模型升级做 eval，不凭“新模型更强”直接替换。
5. 对关键输出保留证据链，避免模型把语言流畅度伪装成正确性。

OpenAI evals 和 LangSmith/Phoenix 这类工具的兴起，本质上也是因为 LLM 应用不能靠单次样例判断质量。模型是概率组件，Agent 是概率组件串成的系统；评估和观测就是把概率系统拉回工程控制面的办法。

## 12. 面试追问视角：可以怎么回答

如果被问“为什么 Agent 不是模型越强越好”，可以答：

> 模型能力决定上限，但 Agent 稳定性取决于上下文、工具、状态、验证和权限。强模型如果拿到噪声上下文、模糊工具 Schema、没有外部验证，也会多步跑偏。

如果被问“大模型和小模型怎么分工”，可以答：

> 常见做法是强模型做规划和复杂推理，小模型做路由、抽取、简单摘要和格式转换。这样既能控制成本，也能让关键步骤获得足够推理能力。

如果被问“LLM 基础对工程最重要的启发是什么”，可以答：

> 它提醒我们模型输出是概率分布，不是确定函数。因此生产 Agent 必须用 eval、trace、schema、verifier 和预算控制来管理不确定性。
