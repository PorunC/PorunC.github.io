Agent 要完成真实任务，不能只靠模型参数里的知识。企业文档、业务规则、最新信息、用户历史、代码仓库、产品手册都不在模型内部，或者已经过期。RAG 的作用，就是把外部知识变成 Agent 每一步可用、可追溯、可更新的证据。

`05_RAG检索增强生成` 模块可以理解成 Agent 的知识供应链：离线把知识整理成索引，在线按任务需要检索，再把证据交给模型生成答案或决策。

## 1. RAG 解决的不是“让模型更聪明”

RAG 的核心价值不是提升模型智商，而是改善事实来源。它把模型从“凭记忆回答”变成“基于证据回答”。

在 Agent 系统中，RAG 主要解决四类问题：

| 问题 | RAG 的作用 |
| --- | --- |
| 私有知识 | 检索企业内部文档、代码、制度 |
| 最新信息 | 通过外部索引或搜索补充新事实 |
| 长文档 | 只取相关片段，避免全塞上下文 |
| 幻觉控制 | 要求答案基于可引用证据 |

但 RAG 不是万能的。检索不到、检索错、上下文拼错、生成越界，都会导致结果失败。RAG 的质量取决于整条链路，而不是只取决于向量数据库。

## 2. 离线构建：知识入库不是简单切文本

离线阶段通常包括清洗、分块、元数据附加、Embedding、入库和建索引。

| 步骤 | 关键决策 | 常见坑 |
| --- | --- | --- |
| 清洗 | 去重、去模板噪声、统一编码 | 垃圾进库导致垃圾检索 |
| 分块 | chunk size、overlap、分隔符 | 太小丢语义，太大带噪声 |
| 元数据 | source、doc_id、章节、时间、权限 | 没来源就无法引用和过滤 |
| Embedding | 模型、维度、语言能力 | 换模型通常要全库重建 |
| 索引 | HNSW、IVF、过滤条件 | 数据规模和延迟要求不匹配 |

分块尤其重要。好的 chunk 不只是固定长度切片，而是尽量保留语义边界：标题、段落、表格、代码块、章节层级。对政策文档，章节结构很重要；对代码，函数和类边界很重要；对 FAQ，问答对边界很重要。

## 3. 在线查询：召回、精排、拼上下文

在线阶段看起来简单：问题向量化，向量库搜索，拿结果给模型。实际效果往往卡在三个地方。

第一是查询表达。用户问题可能很短，也可能带口语、省略、上下文指代。直接 embedding 可能召回不准，所以需要 Query Rewrite，把“这个怎么配置”改成包含当前产品、功能、版本的检索查询。

第二是召回宽度。粗检索 Top-K 太小会漏掉答案，太大又会带来噪声。常见策略是粗检索开大，例如 20 到 50，再用 Rerank 精排到 3 到 5 个片段。

第三是上下文拼接。RAG context 不是把片段简单拼起来。需要带来源编号、标题、时间、权限标识，并尽量去重和排序。

## 4. Embedding 与相似度搜索

Embedding 把文本映射到向量空间，让语义相近的内容距离更近。向量检索擅长“语义相似”，但它不等于“能回答问题”。

例如用户问“如何重置企业账号的 MFA”，向量检索可能召回“账号安全最佳实践”，因为语义接近，但真正答案可能在“管理员后台操作手册”的某个小节。相似不等于支持。

这就是为什么 Rerank 重要。Bi-encoder 向量检索适合大规模粗筛；Cross-Encoder Rerank 把 query 和文档片段一起看，能更精确判断相关性，但成本更高，所以只在候选集上跑。

## 5. 召回率和精确率的平衡

RAG 优化有两个核心方向：召回率和精确率。

| 指标 | 含义 | 优化手段 |
| --- | --- | --- |
| 召回率 | 正确证据是否被找回来 | query rewrite、多路检索、增大 Top-K、混合检索 |
| 精确率 | 返回证据是否真正相关 | rerank、阈值过滤、去重、上下文压缩 |

只追求召回，会把大量无关片段塞给模型，增加幻觉风险。只追求精确，可能漏掉关键证据。更稳的做法是“两阶段”：粗召回尽量不漏，精排和过滤控制最终上下文质量。

## 6. RAG 如何减少幻觉

RAG 减少幻觉的前提是模型真的被约束在证据内。仅仅提供资料，不代表模型不会越界。需要在 Prompt 和后处理里明确：

1. 只基于给定文档回答。
2. 无证据时说不知道。
3. 关键结论必须引用来源编号。
4. 生成后校验答案中的声明是否被来源支持。

对高风险场景，可以加一个 Answer Verifier：输入答案和检索证据，判断每个结论是否有支持。如果没有支持，就要求模型修改或降级回答。

## 7. RAG 与长上下文

长上下文模型让人容易误以为 RAG 不重要。实际上，长上下文解决的是“能不能放进去”，不是“该不该放进去”。把几十万 token 全塞给模型，成本高、延迟高、注意力分散，也不利于权限过滤和来源追踪。

RAG 仍然有价值：

| 能力 | 长上下文 | RAG |
| --- | --- | --- |
| 容纳大量文本 | 强 | 按需取片段 |
| 成本控制 | 弱 | 强 |
| 来源追踪 | 需要额外设计 | 天然适合 |
| 增量更新 | 每次重新输入 | 索引更新即可 |
| 权限过滤 | 复杂 | 可在检索层处理 |

实际系统常用混合方案：RAG 先筛出相关内容，再利用长上下文模型处理更完整的证据集合。

## 8. RAG 和微调如何选择

RAG 适合知识更新频繁、需要引用来源、私有资料多的场景。微调适合改变模型行为风格、输出格式、领域表达习惯，或者让模型学会某类稳定任务模式。

| 需求 | 更适合 |
| --- | --- |
| 公司知识库问答 | RAG |
| 最新政策和产品文档 | RAG |
| 固定格式客服话术 | 微调或 Prompt |
| 专业术语表达风格 | 微调 |
| 回答必须给来源 | RAG |
| 工具调用模式稳定化 | 微调或 Few-shot |

很多生产系统会组合使用：RAG 负责事实，Prompt 或微调负责表达和流程。

## 9. Agentic RAG

普通 RAG 是一次检索一次生成。Agentic RAG 则让 Agent 参与检索过程：它可以判断第一次结果是否足够，改写查询，多跳检索，比较不同来源，必要时继续搜。

这适合复杂问题，但也带来成本和延迟。需要设置最大检索轮数、证据足够性的判断标准和失败策略。

一个成熟的 Agentic RAG 流程通常是：

1. 分析问题需要哪些证据。
2. 生成一个或多个检索查询。
3. 检索并精排候选证据。
4. 判断证据是否足够。
5. 不足则改写查询继续。
6. 足够则生成答案并验证来源。

## 10. RAG 落地最难的地方

RAG 最难的不是把向量库跑起来，而是让知识链路长期保持干净、可更新、可评估。

需要重点关注：

| 难点 | 说明 |
| --- | --- |
| 数据质量 | 源文档重复、过期、冲突会污染结果 |
| 权限过滤 | 多用户、多租户不能串数据 |
| 增量更新 | 文档变化后索引要及时更新 |
| 评估体系 | 要衡量检索命中、答案忠实度、引用正确性 |
| 成本延迟 | embedding、rerank、长上下文都会消耗资源 |

## 11. 这个模块的工程结论

RAG 是 Agent 的知识基础设施。设计时要记住：

1. 离线索引质量决定在线检索上限。
2. Chunking 和元数据不是细节，是可追溯性的基础。
3. 向量相似不等于答案相关，Rerank 很关键。
4. RAG 能降低幻觉，但必须配合引用和验证。
5. 长上下文不是 RAG 的替代品，而是增强工具。
6. Agentic RAG 适合复杂问题，但必须有轮数和成本边界。

如果 Agent 没有可靠知识来源，它就只能猜。如果 RAG 没有评估和治理，它只是把“猜”换成了“从噪声里猜”。真正可用的 RAG，是一条从数据入库到答案验证都可控的证据链。

## 12. 外部资料补强：从 RAG 论文到生产检索系统

Lewis 等人的 RAG 论文明确把模型参数中的知识和外部非参数记忆结合起来，核心价值包括知识更新、来源追踪和更具体的事实生成。这个定义能帮助我们避免把 RAG 简化成“向量库问答”。真正的 RAG 是生成模型和检索系统的组合，检索结果必须以证据形式参与生成。

Dense Passage Retrieval 论文证明，双编码器 dense retrieval 可以在开放域问答中显著提升 passage retrieval 表现。它解释了为什么 embedding 检索成为 RAG 的主流基础。但 dense retrieval 也有天然局限：向量相似表示“语义近”，不保证“能回答”。这正是生产系统常加 rerank、metadata filter、hybrid search 的原因。

HNSW 论文和 Faiss 文档提供了向量检索工程的底层视角：高维近邻搜索需要近似索引，否则数据规模上来后无法低延迟查询。理解这一点后，向量数据库选型就不只是“哪个 API 好用”，还要看索引类型、过滤能力、增量更新、评估工具和运维成本。

RAGAS 提醒我们，RAG 评估至少要分检索相关性、上下文聚焦度、答案忠实度和回答质量。只看最终答案会掩盖问题来源。生产 RAG 应该同时评估 retrieval 和 generation，否则优化方向会跑偏。
