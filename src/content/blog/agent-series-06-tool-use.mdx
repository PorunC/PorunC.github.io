工具调用是 Agent 从“会说”变成“会做”的分界线。没有工具，模型只能基于参数记忆生成文本；有了工具，Agent 可以查数据库、调用 API、搜索网页、读写文件、执行代码、操作业务系统。

但工具也是 Agent 风险最高的部分。模型一旦选错工具、填错参数或越权调用，错误就会从文本世界进入真实系统。因此 `06_工具调用` 模块的核心不是“怎么接一个函数”，而是“怎么把工具设计成模型能正确选择、系统能安全执行、结果能可靠验证的接口”。

## 1. Tool Use 的本质：给模型一组可执行动作

工具不是普通 API 文档。它的读者首先是模型。模型通过工具名称、描述和参数 Schema 判断：这个工具能不能解决当前子问题，什么时候该用，参数怎么填，调用后如何解释返回结果。

一个工具至少包括三部分：

| 字段 | 作用 |
| --- | --- |
| name | 稳定标识，影响模型初筛 |
| description | 告诉模型何时使用、何时不使用 |
| input_schema | 参数类型、必填项、枚举、范围、格式 |

模型看到的不是你的真实函数实现，而是这份抽象合同。合同写得模糊，模型就会误签。

## 2. 工具选择机制

现代 Function Calling 通常由模型在生成时决定是否输出工具调用。它会结合用户任务、系统指令、工具描述和历史 observation，选择工具名并生成参数。

工具选择容易受这些因素影响：

| 因素 | 影响 |
| --- | --- |
| 工具名 | `search_product_catalog` 比 `search` 更清楚 |
| description | 是否说明适用和不适用场景 |
| 参数复杂度 | 参数越多，填错概率越高 |
| 工具数量 | 数量过多会降低选择准确率 |
| 历史示例 | Few-shot 可以显著改善选择 |

经验上，一个 Agent 一次暴露太多工具会变差。工具很多时应分组、检索或路由：先选工具类别，再暴露具体工具。

## 3. 好的 Tool Schema 怎么写

好的 Schema 像给模型看的最小 API 手册。它要自解释、少歧义、可校验。

| 原则 | 示例 |
| --- | --- |
| 名称具体 | `get_user_order_history` 优于 `get_data` |
| 描述说明使用时机 | “适合查询订单状态，不适合查询商品库存” |
| 参数带格式 | `YYYY-MM-DD，例如 2026-06-17` |
| 用 enum 收口 | `status: pending | paid | shipped` |
| 禁止额外参数 | `additionalProperties: false` |
| 参数数量克制 | 高频参数暴露，低频参数默认 |

工具描述里加入负面说明很有用。模型不仅要知道“什么时候用”，还要知道“什么时候不用”。多工具场景里，负面边界能减少误调用。

## 4. Function Calling 是结构化 ReAct

早期 ReAct 依赖文本格式：

```text
Thought: ...
Action: search_web
Action Input: ...
Observation: ...
```

现代 Function Calling 把 `Action` 结构化，模型直接返回工具名和 JSON 参数。这样减少了解析错误，也方便运行时校验。

但 Function Calling 只解决了“格式更可靠”，没有自动解决“选择一定正确”。工具描述、权限、执行结果和错误恢复仍然要靠系统设计。

## 5. 数据库、搜索与代码执行

不同工具类型的风险完全不同。

| 工具 | 关键设计 |
| --- | --- |
| 数据库查询 | 只读优先、参数化查询、租户过滤、行级权限 |
| 搜索引擎 | 查询改写、结果去重、来源可信度、时间过滤 |
| 外部 API | 超时、重试、限流、幂等键 |
| 文件读写 | 路径白名单、大小限制、审计日志 |
| 代码执行 | 沙箱、断网、CPU/内存/时间限制 |

代码执行尤其危险。不要把模型生成的表达式直接 `eval`。生产环境需要白名单表达式求值、AST 限制，或者容器/虚拟机沙箱，并且默认断网、限制文件系统。

## 6. 多工具协调

复杂任务常常需要多个工具。多工具调用有三种常见模式：

| 模式 | 说明 |
| --- | --- |
| 串行调用 | 上一步结果决定下一步 |
| 并行调用 | 多个独立信息源同时查询 |
| 层级调用 | 高层 Agent 把 Worker Agent 当工具 |

并行工具调用能降低延迟，但要求工具之间没有强依赖。串行调用更灵活，但容易增加步数。层级调用适合多 Agent 系统，但会显著增加 trace 和状态管理复杂度。

工具结果也要统一格式。返回一大段自由文本，会让模型自己解析；返回结构化 JSON、摘要、状态码和错误字段，系统才能可靠处理。

## 7. 工具调用失败后的处理

工具失败不要直接让 Agent 崩溃。更好的方式是把错误作为 Observation 返回模型，同时由 Harness 控制重试边界。

| 失败类型 | 处理策略 |
| --- | --- |
| 参数校验失败 | 返回具体字段错误，让模型修正 |
| 网络超时 | 有限重试，指数退避 |
| 权限不足 | 不重试，说明权限边界 |
| 结果为空 | 允许改写查询或换工具 |
| 工具异常 | 记录日志，返回可读错误 |

关键是错误信息要可行动。例如“参数错误”不够，应该告诉模型“date 必须是 YYYY-MM-DD，当前值是 17/06/2026”。

## 8. 工具结果可靠性验证

工具返回不等于可信。搜索结果可能过期，数据库结果可能为空，代码执行可能被截断，外部 API 可能返回部分失败。

验证可以分几层：

1. Schema 验证：返回结构是否符合预期。
2. 业务验证：状态码、字段范围、权限是否正确。
3. 来源验证：结果来自哪里，是否可追溯。
4. 交叉验证：关键事实用多个来源确认。
5. LLM Verifier：判断结果是否足以支持下一步。

在 RAG 或搜索场景里，工具结果最好带来源、时间和置信度。这样最终答案可以引用，审计也能追踪。

## 9. 插件系统与 MCP

当工具数量增加，硬编码工具列表会变得不可维护。插件系统把工具打包成可发现、可加载、可版本化的模块。一个成熟插件通常包含工具 Schema、执行实现、权限声明、配置项和文档。

MCP（Model Context Protocol）可以理解为一种标准化连接外部上下文和工具的协议。它试图解决的问题是：不同 Agent 客户端如何以统一方式发现工具、读取资源、调用能力，而不是每个系统都自定义一套接口。

无论是插件还是 MCP，核心诉求都是工具生态化：工具不再是散落在代码里的函数，而是有描述、有边界、有生命周期的能力包。

## 10. 工具权限控制

工具权限必须独立于模型判断。模型可以请求调用工具，但系统要决定是否允许。

权限控制至少包括：

| 维度 | 示例 |
| --- | --- |
| 用户权限 | 普通用户不能访问管理员工具 |
| 工具风险级别 | 删除、转账、发邮件需要审批 |
| 参数范围 | 只能查询自己的 user_id |
| 调用频率 | 每任务最多调用 N 次 |
| 环境权限 | 代码执行默认断网、只读文件系统 |
| 审计级别 | 高风险工具记录完整入参出参 |

最小权限原则在 Agent 里比传统系统更重要，因为模型可能被注入或误导。即使模型想越权，工具层也必须挡住。

## 11. 这个模块的工程结论

工具调用是 Agent 的行动层。要做稳，需要同时关注模型可理解性和系统安全性：

1. Tool Schema 是给模型看的 API 合同。
2. 工具描述要写“什么时候用”和“什么时候不用”。
3. Function Calling 降低格式错误，但不替代权限和验证。
4. 工具结果要结构化、可追踪、可验证。
5. 失败信息要作为 Observation 返回，让 Agent 可恢复。
6. 工具权限必须由系统强制执行，不能交给模型自觉。
7. 工具多了要插件化、分组化、协议化。

一个 Agent 的真实能力，往往不是模型参数决定的，而是工具层决定的；一个 Agent 的真实风险，也往往从工具层开始。

## 12. 外部资料补强：Tool Use 正在标准化

Toolformer 论文展示了一个重要方向：语言模型可以通过少量示例学习在合适位置调用外部工具，例如计算器、问答系统、搜索引擎和翻译系统。它说明工具使用不是简单“外挂 API”，而是一种可以被模型学习的行为模式。工程系统虽然不一定训练 Toolformer 式模型，但可以借鉴它的思想：工具描述、调用时机和返回结果都要进入模型可学习的上下文。

OpenAI 的 Function Calling 文档把工具调用定义为应用和模型之间的协作流程。模型提出调用，应用执行函数，再把结果交回模型。这个流程明确了责任边界：模型不直接执行外部动作，执行权在应用。也正因为如此，Schema 校验、权限控制、幂等、防重放、审计都必须在应用侧实现。

MCP 官方文档把工具、资源和提示词统一放进一个协议体系里。它的意义在于让 Agent 客户端可以标准化发现外部能力，而不是每个工具都定制一套接入方式。对企业来说，MCP 的价值不是“多一个协议名”，而是把工具接入从临时代码升级成可治理的能力目录。

LangChain 和 LangGraph 的工具文档也反复强调 structured tools、schema 和 runtime context。框架越成熟，越会把工具从“函数指针”提升成“带描述、参数、状态和执行策略的对象”。这正是工具工程化的方向。

## 13. 深度补充：为什么 Tool Use 会成为 Agent 的分水岭

Tool Use 出现的根本原因，是 LLM 的参数记忆和自然语言生成只能解决“知道”和“表达”，很难单独解决“确认、计算、行动、更新”。一个模型可以解释退款政策，但它不知道某个用户的真实订单状态；它可以写 SQL，但不应该直接拿生产库连接字符串；它可以推断今天可能下雨，但不能替你读取公司内部排班系统。

工具调用把 Agent 从封闭文本系统变成开放执行系统。它解决四个基础问题：

| 问题 | 没有工具时的表现 | 有工具后的变化 |
| --- | --- | --- |
| 信息新鲜度 | 只能依赖训练数据或上下文 | 查询实时数据库、搜索、文件、API |
| 精确计算 | 容易心算错误或编造 | 调用计算器、代码解释器、规则引擎 |
| 外部行动 | 只能建议用户怎么做 | 创建工单、发通知、改配置、跑测试 |
| 环境反馈 | 无法验证执行是否成功 | 通过 observation 继续修正计划 |

这也是为什么工具调用一旦进入 Agent，系统设计立刻变复杂：工具把模型的错误从“说错”放大成“做错”。所以 Tool Use 的核心不是把函数暴露出去，而是把函数变成可理解、可约束、可审计的行动接口。

## 14. 应用方式：工具不是一类东西，而是多种能力边界

真实业务里，工具可以按风险和语义分层，而不是一股脑都给模型。

| 工具类型 | 典型应用 | 设计重点 |
| --- | --- | --- |
| 查询型工具 | 查订单、查库存、查日志、查知识库 | 只读、租户过滤、结果分页、来源可追溯 |
| 计算型工具 | 表格计算、指标计算、代码运行、模拟推演 | 输入校验、可重复、限制 CPU/内存/时间 |
| 操作型工具 | 创建工单、发送邮件、修改配置、发起退款 | 幂等键、审批、回滚、审计 |
| 感知型工具 | 浏览器、文件读取、截图、OCR、网页搜索 | 内容不可信、大小限制、注入隔离 |
| 编排型工具 | 把子 Agent、工作流或 MCP server 当工具 | 状态传递、预算隔离、trace 合并 |

客服 Agent 通常优先暴露查询型工具，例如订单状态、物流轨迹、售后政策。只有当规则明确且风险可控时，才暴露操作型工具，例如创建售后单。代码 Agent 则更依赖感知型和计算型工具：搜索文件、读 diff、运行测试、解析日志。研究 Agent 则需要搜索、RAG、引用管理和事实核验工具。

一个实用原则是：工具越接近真实世界副作用，越要窄、越要可审计、越要有审批。不要把 `execute_any_api`、`run_shell_command`、`query_database` 这种万能能力直接交给模型。看起来灵活，实际上把权限、参数、业务规则全部甩给了最不稳定的一层。

## 15. 为什么 Schema 是合同，而不是注释

很多系统把 Tool Schema 当成文档注释，这是危险的。对 Agent 来说，Schema 至少承担三份合同。

第一，它是模型选择工具的语义合同。工具名、description、参数名会影响模型是否认为它适合当前任务。Anthropic 关于工具设计的文章也强调，工具定义要站在模型视角看：如果人类读了都要想半天，模型大概率也会误用。

第二，它是运行时校验的结构合同。参数类型、枚举、范围、必填项、额外字段限制，决定了执行器能不能在真正调用外部系统前拦住错误。

第三，它是审计和演化的版本合同。工具改名、参数改语义、返回结构变更，都会影响 Agent 行为。成熟系统应该记录 tool version、schema version、prompt version 和 model version，否则线上回放时很难解释“为什么上周能用，这周失败”。

好的工具合同通常会包含：

| 设计项 | 例子 |
| --- | --- |
| 正向使用场景 | “用于查询当前登录用户的订单列表” |
| 负向边界 | “不能用于查询其他用户订单，不能发起退款” |
| 参数格式 | `start_date` 必须为 `YYYY-MM-DD` |
| 业务约束 | `limit` 最大 50，默认按最近更新时间倒序 |
| 错误语义 | `NOT_FOUND`、`PERMISSION_DENIED`、`RATE_LIMITED` |
| 返回来源 | 数据库表、文档 URL、检索片段 id |

## 16. 反模式：工具越多，Agent 不一定越强

工具系统最常见的反模式，是把“能接入更多 API”误解成“Agent 更强”。实际表现往往相反。

| 反模式 | 结果 | 修正方式 |
| --- | --- | --- |
| 一次暴露几十个工具 | 模型选择困难，误调用上升 | 先路由工具类别，再暴露子集 |
| 工具描述只写功能 | 模型不知道何时不用 | 写清适用、不适用、边界和例子 |
| 返回大段自由文本 | 模型解析不稳定 | 返回结构化 JSON、摘要和来源 |
| 工具太通用 | 权限和参数难治理 | 拆成窄工具，业务规则前置 |
| 没有错误语义 | 失败后只会重复调用 | 返回可修复错误和下一步建议 |
| 没有 eval | 工具改了才发现 Agent 退化 | 建工具调用评估集和 trace 回归 |

Anthropic 的工具设计实践里有一个很朴素但重要的观点：更少、更清晰、更贴近任务的工具，通常比一堆泛化工具更可靠。生产系统可以把底层 API 很丰富，但模型当前轮次看到的工具应该是经过 Harness 筛选后的最小集合。

## 17. 面试追问视角：可以怎么回答

如果被问“Function Calling 已经结构化了，为什么还会失败”，可以答：

> Function Calling 只解决输出格式问题，不解决工具选择是否正确、参数业务语义是否正确、工具是否越权、返回结果是否可信。真正稳定的 Tool Use 需要 Schema、权限、执行器、错误 observation、trace 和 eval 一起工作。

如果被问“为什么需要 MCP”，可以答：

> MCP 解决的是工具和外部上下文的标准化接入问题。没有协议时，每个 Agent 客户端都要定制工具发现、资源读取、参数格式和连接方式。MCP 把 Tools、Resources、Prompts 等能力放进统一模型里，方便生态复用，但它不自动解决权限、安全和质量评估。

如果被问“如何设计一个查订单工具”，可以答：

> 我会把身份放在系统上下文里，而不是让模型传 `user_id`；工具只查当前用户订单；参数用 enum 和日期格式收口；返回结构化订单状态、来源、更新时间；失败时区分找不到、无权限、超时；高风险操作如退款不放在同一个工具里，而是单独工具加审批。
