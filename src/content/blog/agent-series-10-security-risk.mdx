Agent 安全比普通 LLM 应用更难，因为 Agent 不只是输出文本，它能调用工具、访问数据、执行代码、影响外部系统。攻击者一旦诱导模型做错事，后果可能从“回答不准”升级成“数据泄露、越权操作或成本攻击”。

`10_安全与风险` 模块的核心，是把 Agent 当成一个有行动能力的软件系统来做威胁建模，而不是把安全问题简化成“模型要听话”。

## 1. Agent 的攻击面

Agent 的攻击面来自四个方向：

| 攻击面 | 示例 |
| --- | --- |
| 输入 | Prompt 注入、恶意长文本、越狱请求 |
| 上下文 | RAG 文档投毒、记忆投毒、工具结果注入 |
| 工具 | 越权调用、高风险操作、参数篡改 |
| 输出 | 泄露 PII、系统提示词、密钥、内部数据 |

普通聊天机器人被诱导后可能说错话；Agent 被诱导后可能删文件、发邮件、查数据库、执行代码。因此安全必须进入工具层、数据层和运行时。

## 2. STRIDE 威胁建模

Agent 安全策略可以用 STRIDE 梳理：

| 威胁 | Agent 场景 |
| --- | --- |
| Spoofing 欺骗 | 伪造用户身份访问他人数据 |
| Tampering 篡改 | 记忆投毒、RAG 文档投毒 |
| Repudiation 抵赖 | 无审计日志，无法追责 |
| Information Disclosure 信息泄露 | 输出 PII、系统 prompt、密钥 |
| Denial of Service 拒绝服务 | 超长输入、无限循环、成本耗尽 |
| Elevation of Privilege 权限提升 | 注入后调用高权限工具 |

这比零散地想“要防 prompt 注入”更全面。Prompt 注入只是其中一种输入攻击，真正的安全体系要覆盖身份、权限、数据、工具、日志、恢复。

## 3. Prompt 注入与越狱

Prompt 注入是攻击者把恶意指令混入用户输入、网页、文档或工具返回里，试图覆盖系统指令。越狱则更偏诱导模型绕过安全规则。

防御思路：

1. 指令层级清楚：System > Developer > User > External Data。
2. 外部内容标记为数据，不能当作指令执行。
3. 对高风险短语和行为做检测。
4. 工具层做强制权限，不信模型自觉。
5. 输出层扫描敏感信息。

最重要的是第三方内容隔离。网页、邮件、PDF、RAG 文档都可能包含“忽略之前指令”之类文本。Agent 应该把它们当被分析对象，而不是命令来源。

## 4. 数据泄露风险

Agent 可能在多个位置泄露数据：

| 位置 | 风险 |
| --- | --- |
| Prompt | 把敏感上下文发给不该看的模型或用户 |
| RAG | 检索到其他租户文档 |
| Memory | 记住并复用用户隐私 |
| Logs | 日志里保存密钥、PII、工具结果 |
| Output | 最终答案包含敏感信息 |
| Tools | 查询接口缺少权限过滤 |

多租户系统里，RAG 检索必须按 tenant_id 或 user_id 过滤。不能先全库检索再让模型“不要用别人的数据”，因为泄露已经发生在上下文注入阶段。

## 5. 权限提升与工具权限

权限提升攻击的典型形式是：模型本来只能读数据，但被诱导调用写操作；本来只能查当前用户，却传入别人的 user_id；本来不能发邮件，却通过某个通用 API 绕过去。

工具权限要分层：

| 层级 | 控制 |
| --- | --- |
| 用户权限 | 用户能不能使用这个 Agent |
| 工具权限 | Agent 能不能调用这个工具 |
| 参数权限 | 参数是否在允许范围内 |
| 数据权限 | 查询结果是否属于当前租户 |
| 操作审批 | 高风险动作是否需要人工确认 |

不要给 Agent 一个万能 `execute_api` 工具。工具越通用，越难做权限边界。生产系统更适合暴露窄工具、白名单参数和明确审批点。

## 6. 多用户数据隔离

多用户隔离必须在存储和检索层完成，而不是只靠应用层约定。

常见措施：

1. 数据库行级权限或 tenant_id 强过滤。
2. 向量库 namespace 按租户或用户隔离。
3. Redis key 带租户前缀和 TTL。
4. 日志脱敏并按权限访问。
5. Memory 按用户分区，禁止跨用户召回。
6. 工具调用自动注入当前身份，不让模型自由填写。

模型生成的 `user_id` 不可信。用户身份应该由系统上下文传入工具层，而不是由模型参数决定。

## 7. 代码执行沙箱

代码执行是高风险工具。安全沙箱至少要控制：

| 资源 | 控制方式 |
| --- | --- |
| CPU | 时间限制 |
| 内存 | cgroup 或容器限制 |
| 文件系统 | 临时目录、只读挂载 |
| 网络 | 默认断网或域名白名单 |
| 进程 | 禁止 fork bomb |
| 密钥 | 不注入生产环境变量 |

沙箱输出也要限制大小，防止模型生成代码打印大量内容撑爆上下文。

## 8. 恶意输入与内容安全

恶意输入不只是注入，还包括超长 prompt、重复字符、编码混淆、敏感内容、违法请求、诱导输出隐私等。

输入层可以做：

1. 长度限制。
2. 文件类型和大小限制。
3. 注入模式检测。
4. 敏感类别检测。
5. 速率限制。
6. 任务风险分类。

输出层可以做：

1. PII 脱敏。
2. 密钥和 token 检测。
3. 系统提示词泄露检测。
4. 高风险内容拦截。
5. 引用来源检查。

## 9. 审计日志

安全系统必须能回答：谁，在什么时候，给了什么输入，Agent 看到了什么上下文，调用了什么工具，传了什么参数，拿到了什么结果，最终输出了什么。

审计日志应包含：

| 字段 | 说明 |
| --- | --- |
| trace_id | 请求链路 |
| user_id / tenant_id | 身份和租户 |
| agent_version | 复现行为 |
| prompt_version | 复现提示策略 |
| model | 模型版本 |
| tool_name / args | 工具调用 |
| approval_id | 高风险审批记录 |
| redaction_status | 是否脱敏 |
| risk_level | 风险等级 |
| final_status | 成功、拒绝、降级、失败 |

审计日志要脱敏和权限控制。日志本身也是敏感数据。

## 10. 安全策略的四层架构

Agent 安全策略可以分四层：

| 层 | 目标 |
| --- | --- |
| Prevention 预防 | 最小权限、输入校验、沙箱、密钥管理 |
| Detection 检测 | 异常行为、内容安全、权限异常、成本异常 |
| Response 响应 | 熔断、停止任务、冻结用户、人工介入 |
| Recovery 恢复 | 数据恢复、规则更新、复盘、补测试 |

预防不可能 100% 成功，所以检测、响应和恢复同样重要。生产系统要有 kill switch，能停止失控 Agent 或禁用高风险工具。

## 11. 这个模块的工程结论

Agent 安全的底层原则是：模型输出不可信，外部内容不可信，工具参数不可信。

1. 外部内容只能作为数据，不能作为指令。
2. 权限必须在工具和数据层强制执行。
3. 多租户隔离要发生在检索前，而不是生成后。
4. 高风险工具需要审批、审计和回滚。
5. 代码执行默认沙箱、断网、限资源。
6. 输入和输出都要安全过滤。
7. 安全策略要配置化、版本化、持续红队测试。

一个真正安全的 Agent，不是永远不会被诱导，而是即使模型被诱导，系统边界也不会被突破。

## 12. 外部资料补强：安全从 Prompt 层进入系统治理

OWASP Top 10 for LLM Applications 把 Prompt Injection、Sensitive Information Disclosure、Supply Chain、Excessive Agency 等列为核心风险。其中 Excessive Agency 对 Agent 尤其关键：如果系统给模型过多权限、过宽工具、过高自主性，即使没有传统漏洞，也可能因为模型误判或被诱导而造成真实损害。

NIST AI RMF 提供了更高层的治理框架，强调风险识别、测量和管理。对 Agent 系统来说，安全不应只在上线前做一次检查，而要进入持续流程：红队测试、审计日志、事件响应、策略版本管理、依赖扫描和评估集更新。

OpenAI 和 Anthropic 的工具/agent 文档都强调应用侧执行工具、应用侧控制权限。这一点很重要：模型只是生成请求，不能成为安全边界。真正的安全边界必须在 API Gateway、Tool Executor、Policy Engine、Sandbox、Data Layer 中实现。

因此，Agent 安全正在从“prompt 防注入技巧”升级为完整安全工程：威胁建模、最小权限、隔离、审计、运行时策略、输出过滤、事件响应缺一不可。
