场景题是 Agent 知识体系里最接近真实工作的部分。前面模块讲概念、架构、RAG、工具、评估、安全；场景题要求你把这些能力组合起来，面对一个业务目标做系统设计。

`11_场景题` 模块的价值，不是背“智能客服 Agent 怎么答”，而是训练一种分析方法：先定义目标和边界，再选知识来源、工具、执行流程、评估指标、安全策略和降级方案。

## 1. 场景题的通用答题框架

不管是客服、报告、金融、代码生成，通用框架都类似：

| 步骤 | 要回答的问题 |
| --- | --- |
| 目标 | Agent 要完成什么，不做什么 |
| 输入 | 用户提供什么，系统可感知什么 |
| 知识 | 需要哪些文档、数据库、实时信息 |
| 工具 | 需要查询、计算、操作哪些系统 |
| 流程 | 单 Agent、DAG、多 Agent 还是混合 |
| 记忆 | 是否需要用户偏好、历史任务、长期经验 |
| 安全 | 权限、隐私、审批、审计怎么做 |
| 评估 | 如何判断成功，如何发现失败 |
| 运维 | 延迟、成本、降级、监控怎么处理 |

这个框架比死记具体方案更有用。面试或系统设计时，对方真正想看的是你能不能把 Agent 放进业务约束里。

## 2. 报告撰写 Agent

报告撰写 Agent 的难点不是“写得通顺”，而是资料可靠、结构清楚、引用准确、能迭代修改。

典型流程：

1. 明确报告主题、读者、长度、风格和截止时间。
2. 拆解研究问题。
3. 多源检索资料。
4. 筛选可信来源。
5. 生成大纲。
6. 撰写正文。
7. 做引用和事实核验。
8. 输出最终格式。

适合用多 Agent 或多节点工作流：Researcher 负责检索，Analyst 负责提炼，Writer 负责成文，Reviewer 负责事实和结构检查。

关键指标包括引用准确率、事实错误率、结构完整度、用户修改率、生成成本。

## 3. 智能客服 Agent

客服 Agent 要优先稳定和安全，而不是炫技。它通常需要 RAG、用户订单系统、工单系统、人工转接和内容安全。

架构可以是：

| 模块 | 作用 |
| --- | --- |
| 意图识别 | 判断咨询、投诉、订单、退款、故障 |
| RAG | 查询 FAQ、政策、产品文档 |
| 订单工具 | 查询当前用户订单 |
| 工单工具 | 创建或更新工单 |
| 人工转接 | 高风险或低置信度时接管 |
| 输出校验 | 防止承诺超权限事项 |

客服场景最重要的是权限：Agent 只能查当前用户的数据，不能根据模型生成的 user_id 查询。退款、赔付、改地址等操作应有审批或明确业务规则。

## 4. 金融分析 Agent

金融分析 Agent 风险高，必须区分“信息整理”和“投资建议”。它需要数据来源追踪、时间戳、风险提示和合规限制。

核心设计：

1. 市场数据工具提供价格、财报、公告、宏观数据。
2. RAG 检索研报、公告和历史资料。
3. 计算工具做指标和估值。
4. 输出必须标明数据时间和来源。
5. 禁止给出确定性收益承诺。
6. 高风险建议需要人工或合规审核。

金融 Agent 的评估不能只看文本质量，还要看数据准确性、引用来源、风险披露和合规性。

## 5. 代码生成 Agent

代码生成 Agent 的核心不是生成代码，而是能在仓库环境里理解、修改、测试和回滚。

典型流程：

1. 读取需求和仓库结构。
2. 检索相关文件。
3. 制定修改计划。
4. 编辑代码。
5. 运行测试和静态检查。
6. 根据错误修复。
7. 总结变更和风险。

工具包括文件读写、代码搜索、测试运行、包管理、Git diff。安全边界包括禁止破坏用户未授权文件、谨慎执行命令、记录所有改动。

代码 Agent 的成功标准应包括测试通过、变更范围小、符合项目风格、没有引入安全问题。

## 6. 多 Agent 协作系统

多 Agent 场景适合任务天然分工，例如报告、代码审查、复杂调研、产品分析。核心问题是协调。

设计时要明确：

| 问题 | 说明 |
| --- | --- |
| 角色 | 每个 Agent 的职责是否独立 |
| 输入输出 | 每个角色交付什么结构 |
| 共享状态 | 结果写入哪里 |
| 冲突 | 多个结论不一致谁裁决 |
| 预算 | 每个角色 token 和时间上限 |
| 终止 | 何时停止协作 |

多 Agent 不是越多越好。两个清晰角色通常比五个模糊角色更稳。

## 7. 多目标冲突

真实任务常有冲突：质量 vs 速度，准确性 vs 成本，用户体验 vs 合规，自动化 vs 安全。Agent 不能自己凭感觉决定优先级。

应该把目标优先级显式化：

| 冲突 | 常见优先级 |
| --- | --- |
| 安全 vs 完成任务 | 安全优先 |
| 合规 vs 用户满意 | 合规优先 |
| 准确性 vs 速度 | 按场景设置阈值 |
| 成本 vs 质量 | 高价值任务可放宽预算 |
| 自动执行 vs 人工审批 | 高风险操作审批 |

多目标决策最好由业务策略配置，而不是每次让模型临场发挥。

## 8. 长期记忆设计

长期记忆适合保存稳定、可复用的信息：用户偏好、历史任务摘要、成功模式、失败经验。不要保存临时工具结果或敏感数据。

记忆写入要有门槛：

1. 是否长期有效。
2. 是否与用户或业务相关。
3. 是否经过确认。
4. 是否包含隐私或敏感信息。
5. 是否需要过期时间。

记忆召回也要可控。错误记忆会污染后续任务，因此需要编辑、删除、审计和冲突处理。

## 9. 错误排查方法

Agent 输出错误时，不要只看最终答案。按链路排查：

1. 用户目标是否理解错。
2. Prompt 是否给错约束。
3. RAG 是否召回正确证据。
4. 工具是否选对。
5. 工具参数是否正确。
6. 工具返回是否被正确解读。
7. 计划是否跑偏。
8. 最终生成是否越界。

这就是为什么 trace 必须完整。没有中间轨迹，只能猜。

## 10. 成本、延迟与稳定性优化

场景题经常追问“成本过高怎么办”“延迟过高怎么办”“稳定性怎么提升”。回答要分层：

| 问题 | 策略 |
| --- | --- |
| 成本高 | 模型路由、缓存、减少上下文、控制步数、批处理 |
| 延迟高 | 并行工具、流式输出、异步任务、减少 RAG 和 rerank 宽度 |
| 稳定性差 | 重试、熔断、降级、评估集、灰度发布 |
| RAG 效果差 | 查召回、chunk、query rewrite、rerank、数据质量 |
| 成功率低 | 失败归因、改工具 schema、改 planner、加 verifier |

## 11. 生产级 Agent 设计

生产级 Agent 至少要具备：

1. API 认证和限流。
2. 输入输出安全过滤。
3. 工具权限和审计。
4. 状态存储和任务恢复。
5. 结构化日志、指标、trace。
6. 评估集和 A/B 测试。
7. 失败降级和人工接管。
8. 成本和 quota 控制。

Demo 只证明“能跑”，生产系统要证明“可控”。

## 12. 这个模块的工程结论

场景题的核心是组合能力：

1. 先问业务目标和成功标准，再谈模型和框架。
2. 每个场景都要定义知识、工具、流程、安全和评估。
3. 高风险场景优先合规、权限和审计。
4. 多 Agent 只有真实分工清楚时才使用。
5. 错误排查必须基于 trace，而不是看最终答案猜。
6. 成本、延迟、稳定性是设计的一部分，不是上线后的补丁。

如果说前十个模块是零件，场景题就是装配。能把零件按业务约束装成系统，才是真正的 Agent 工程能力。

## 13. 外部资料补强：场景设计要从“可验证任务”开始

Anthropic 在《Building Effective AI Agents》中把 customer support 和 coding agents 作为两个典型落地场景。它们有共同点：任务需要对话加行动，有工具可以接入，有明确成功标准，并且可以通过用户解决率或自动化测试来验证结果。这解释了为什么客服和代码 Agent 比“万能个人助理”更容易生产化。

Anthropic 的客服场景指南也强调，支持类 Agent 要先定义理想交互、拆分独立任务、建立成功标准，再补充工具和动态能力。这和本篇的场景题框架一致：不要从“我要做客服 Agent”直接跳到模型和框架，而要先拆出 FAQ、订单查询、退款、投诉升级、人工转接等子流程。

Codex 的数据分析与报告用例则展示了报告类任务的另一条原则：Agent 产物应该可复现、可审查，而不是一次性聊天结果。对于报告撰写、数据分析、代码修改这类场景，最好让 Agent 生成脚本、引用、diff、测试结果或中间表，而不只是最终文字。

因此，判断一个场景是否适合 Agent，可以用三问：它是否需要语义判断，是否能通过工具获得 ground truth，是否有可验证的成功标准。三者越清楚，Agent 越容易落地。

## 14. 深度补充：场景题真正考的是“约束建模”

场景题表面问“设计一个客服 Agent / 报告 Agent / 代码 Agent”，本质是在问你能不能把业务约束翻译成 Agent 系统边界。模型能力只是其中一部分，更重要的是：哪些事情能自动做，哪些必须人工确认，哪些数据能看，哪些动作不能做，成功和失败怎么定义。

一个场景要能落地，至少要回答五个为什么：

| 问题 | 设计含义 |
| --- | --- |
| 为什么需要 Agent | 是否真的需要动态决策，还是普通 workflow 就够 |
| 为什么需要这些工具 | 工具是否能提供 ground truth 或真实行动能力 |
| 为什么这样拆流程 | 哪些步骤确定，哪些步骤交给模型判断 |
| 为什么这样控风险 | 权限、审批、审计是否匹配业务风险 |
| 为什么这样评估 | 成功标准是否可量化、可回放、可回归 |

这也是为什么“万能 Agent”很难回答。越泛化，目标越模糊，工具越多，权限越难收敛，评估越不可控。反而是客服、代码、报表、数据分析这些具体场景更容易生产化，因为它们有明确输入、工具、边界和验证方式。

## 15. 应用方式一：客服 Agent 的端到端拆法

客服 Agent 是最适合讲清楚“为什么”的场景。它出现不是为了让机器人更会聊天，而是为了解决人工客服成本高、FAQ 覆盖不稳定、订单问题需要查系统、跨渠道体验不一致的问题。

一个稳妥的客服 Agent 可以按四层设计：

| 层 | 设计 |
| --- | --- |
| 意图层 | 售前咨询、订单查询、退款、投诉、技术支持、人工转接 |
| 知识层 | FAQ、政策文档、产品文档、历史工单、用户订单 |
| 工具层 | 订单查询、物流查询、工单创建、退款申请、转人工 |
| 策略层 | 低风险自动回答，高风险审批或转人工 |

关键不在“回答像人”，而在“回答和动作都符合业务规则”。比如退款不是模型觉得可以退就退，而是工具层根据订单状态、时间、商品类型、用户身份、政策版本计算可选动作。模型负责解释和引导，规则/工具负责判定和执行。

一个常见落地流程是：

1. 识别用户意图和风险等级。
2. 如果是政策类问题，先 RAG 检索最新政策。
3. 如果涉及个人订单，工具层用当前登录身份查询，不让模型填写 user_id。
4. 如果需要退款、赔付、改地址，先判断业务规则和审批要求。
5. 如果证据不足、用户情绪升级或风险高，转人工并附带摘要。
6. 记录 trace、引用政策版本、工具调用和最终处理结果。

## 16. 应用方式二：代码 Agent 为什么能比通用办公 Agent 更早落地

代码 Agent 的优势在于环境反馈非常强。测试、类型检查、lint、构建、diff、Git 历史都能提供 ground truth。Agent 不需要只靠“感觉代码对不对”，它可以运行测试，看失败日志，再修复。

这解释了为什么 SWE-bench、coding agent、Codex 类产品会成为 Agent 的代表场景：代码任务天然具备可验证闭环。

<Mermaid size="wide" diagram={`
flowchart LR
  Issue["需求 / Bug 描述"] --> Search["检索相关文件"]
  Search --> Plan["制定修改计划"]
  Plan --> Edit["编辑代码"]
  Edit --> Test["运行测试 / 静态检查"]
  Test --> Result{"通过?"}
  Result -- "否" --> Diagnose["读取错误日志并归因"]
  Diagnose --> Edit
  Result -- "是" --> Review["生成 diff 总结和风险说明"]
`} />

代码 Agent 的安全边界也更清楚：不破坏用户未授权修改，不执行危险命令，不泄露仓库机密，不越过测试失败强行声称完成。它的评估也更客观：测试通过率、patch 接受率、人工 review 通过率、回归率、平均修复步数。

## 17. 应用方式三：报告 Agent 的核心是证据链

报告 Agent 经常被误解成“让模型写一篇文章”。真正有价值的报告 Agent 应该解决的是资料搜集、来源筛选、结构组织、引用追踪和事实核验。

它的设计重点是证据链：

| 环节 | 关键问题 |
| --- | --- |
| 资料搜索 | 来源是否权威，是否足够新 |
| 资料筛选 | 是否去重，是否排除低质量来源 |
| 大纲生成 | 是否覆盖用户关心的问题 |
| 正文撰写 | 每个关键结论是否有来源支撑 |
| 事实核验 | 数字、日期、引用是否一致 |
| 格式输出 | 是否符合目标读者和交付模板 |

报告类任务适合 Researcher/Writer/Reviewer 分工，但前提是每个角色有结构化交付物。Researcher 输出来源表，Writer 输出草稿，Reviewer 输出 claim-level 检查结果。否则多 Agent 只会增加聊天成本。

## 18. 场景题的反模式

| 反模式 | 为什么危险 | 更好的回答 |
| --- | --- | --- |
| 一上来选框架 | 忽略业务目标和约束 | 先定义目标、边界、数据、工具 |
| 什么都用多 Agent | 增加成本和不确定性 | 只有真实分工才多 Agent |
| 把权限交给模型 | 模型可能被诱导或误判 | 工具层强制权限和审批 |
| 没有评估指标 | 无法证明有效 | 任务成功率、轨迹、成本、安全指标 |
| 没有降级方案 | 线上失败不可控 | 人工接管、缓存、只读建议、暂停任务 |
| 忽略数据来源 | 容易幻觉和过期 | 引用、时间戳、来源可信度 |

## 19. 面试追问视角：可以怎么回答

如果被问“如何判断一个场景适不适合 Agent”，可以答：

> 看三点：是否需要语义判断和动态路径，是否能通过工具获得外部事实或执行动作，是否有可验证成功标准。如果只是固定流程和结构化输入，传统 workflow 更稳；如果路径不确定、需要工具反馈、结果可评估，Agent 更合适。

如果被问“设计客服 Agent 的核心风险是什么”，可以答：

> 核心是权限、合规和错误承诺。订单查询必须基于系统身份，退款赔付不能靠模型决定，高风险和低置信度要转人工，回答要引用政策版本，所有工具调用和输出要有审计。

如果被问“为什么代码 Agent 更容易评估”，可以答：

> 因为代码环境能提供强反馈：测试、构建、lint、类型检查、diff 和运行错误。Agent 可以在环境反馈中迭代，评估也能用测试通过率、patch 接受率和人工 review 结果量化。
