## 1. Python 基础为什么和 Agent 有关

Agent 系统大量依赖 Python 的数据结构和运行时特性：

| 知识点 | Agent 中的体现 |
| --- | --- |
| list / tuple / dict | 消息列表、工具注册表、状态对象 |
| 可变对象 | 状态被多个步骤共享时容易误改 |
| 浅拷贝 / 深拷贝 | 回滚、分支计划、并发状态隔离 |
| *args / **kwargs | 工具参数转发 |
| 装饰器 | 注册工具、埋点、权限检查 |
| asyncio | 并发模型调用和工具调用 |
| GIL | CPU 密集任务不能靠线程解决 |

很多 Agent bug 并不是模型问题，而是状态对象被错误复用、异步任务共享了可变 dict、工具参数没有校验。

## 2. 状态管理与拷贝语义

Agent 每一步都在更新 state：当前计划、已完成步骤、工具结果、token 消耗、错误记录。如果状态对象被多个分支共享，就可能产生隐蔽 bug。

例如多 Agent 并行时，如果两个 worker 修改同一个 dict，结果会互相污染。需要明确：

1. 哪些状态只读。
2. 哪些状态由单个步骤写。
3. 并发分支是否需要 deep copy。
4. 合并结果时冲突怎么处理。

Python 的可变对象很方便，但 Agent 状态最好用 dataclass、TypedDict 或 Pydantic 定义结构，减少随意写字段。

## 3. 装饰器与工具注册

装饰器很适合做 Tool Registry。开发者写普通函数，再用 `@tool` 声明名称、描述、参数 Schema、权限级别。

工具注册系统通常要保存：

| 字段 | 用途 |
| --- | --- |
| name | 模型调用标识 |
| description | 工具选择依据 |
| schema | 参数校验 |
| handler | 实际执行函数 |
| permissions | 权限和风险等级 |
| timeout | 执行上限 |

装饰器还可以用于埋点、重试、脱敏和审计。这样工具实现和治理逻辑不会散落各处。

## 4. asyncio 与并发

Agent 里很多任务是 IO 密集型：调用 LLM API、查询向量库、访问数据库、搜索网页、调用外部服务。`asyncio` 可以显著降低总延迟。

适合并发的场景：

1. 多个独立搜索查询。
2. 多个 LLM 评审器打分。
3. 多个工具无依赖调用。
4. 批量 embedding。
5. 多用户请求并发处理。

但并发要有限制。需要 semaphore 控制最大并发，避免触发 API rate limit。

GIL 对 IO 密集影响不大，但 CPU 密集任务，例如本地 embedding、大规模 Pandas 处理、代码分析，可能需要 multiprocessing、向量化库或独立服务。

## 5. 批量处理与吞吐

Agent 系统常有批处理需求：批量摘要、批量打标签、批量 embedding、批量评估。单条请求循环调用 API 会很慢，也浪费网络往返。

优化手段：

| 手段 | 说明 |
| --- | --- |
| batch API | 离线任务降低成本 |
| encode_batch | 批量 embedding |
| asyncio.gather | 并发 IO |
| 队列 worker | 平滑处理任务 |
| 结果缓存 | 避免重复处理 |
| backpressure | 下游慢时限制生产速度 |

吞吐优化要结合失败恢复：批量任务中单条失败不应导致整批失败。

## 6. 大文本与 Pandas

Agent 常处理文档、日志、表格、CSV、Excel。Python 工程里要注意内存和分块。

大文本处理适合流式读取、按段分块、增量写入。Pandas 适合结构化数据清洗、聚合、过滤，但大文件要用 chunksize 或更合适的引擎。

对 Agent 来说，工具层应把大结果加工成模型可消费的摘要，而不是把百万行表格直接塞进上下文。

## 7. 余弦相似度与向量检索

RAG 最小实现通常要理解余弦相似度：衡量两个 embedding 方向是否相近。工程上，自己实现适合学习，小规模可以用 numpy，大规模应交给向量数据库。

需要注意：

1. 离线和在线 embedding 模型必须一致。
2. 向量要归一化或明确距离度量。
3. 元数据过滤和向量相似同样重要。
4. 近似索引会带来召回和速度权衡。

## 8. LLM API 客户端

生产级 LLM client 不应只是 `client.chat(...)`。它至少要封装：

| 能力 | 说明 |
| --- | --- |
| 超时 | 防止请求挂死 |
| 重试 | 处理 429、5xx、网络错误 |
| 指数退避 | 避免雪崩重试 |
| 模型路由 | 根据任务选模型 |
| token 统计 | 成本监控 |
| 流式输出 | 改善体验 |
| 日志脱敏 | 保护隐私 |

把这些封装成统一接口，后续换模型提供商或做 fallback 才不会改满项目。

## 9. Streaming

流式输出对用户体验很重要。长任务可以边生成边返回，也可以先返回进度事件：正在检索、正在分析、正在调用工具、正在生成答案。

Streaming 要处理几个细节：

1. 前端断开时是否取消后台任务。
2. 工具调用期间如何发送进度。
3. 输出是否需要增量安全过滤。
4. 最终结果和中间 token 如何区分。
5. trace 如何记录完整响应。

## 10. ReAct 与 Function Calling 实现

Python 实现 ReAct 的骨架包括：

1. messages 保存对话和工具结果。
2. tools 保存模型可见 Schema。
3. registry 保存工具名到函数。
4. loop 控制 max_steps。
5. tool_call 执行前校验参数和权限。
6. observation 追加回 messages。
7. final answer 结束。

文本解析式 ReAct 容易出错，现代实现应优先使用 Function Calling。即便如此，也要保留 max_steps、错误 observation 和工具结果截断。

## 11. RAG Pipeline 实现

完整 RAG Pipeline 可以拆成离线和在线。

离线：

1. 读取文档。
2. 清洗和分块。
3. 添加元数据。
4. 批量 embedding。
5. upsert 向量库。

在线：

1. 查询改写。
2. 查询 embedding。
3. 向量检索。
4. rerank。
5. 拼接 context。
6. LLM 生成。
7. 引用和忠实度校验。

Python 代码要把每一步做成独立函数，方便评估和替换。不要把整个 RAG 写成一个巨大的 `answer_question()`。

## 12. 多轮对话管理

多轮对话不等于把所有历史全塞进去。需要管理：

| 内容 | 策略 |
| --- | --- |
| 最近对话 | 保留原文 |
| 早期历史 | 摘要压缩 |
| 用户偏好 | 写入长期记忆 |
| 当前任务状态 | 单独 state store |
| 无关闲聊 | 不进入关键上下文 |

对话管理器要根据 token 预算动态拼上下文，而不是简单 `messages.append()` 到无限长。

## 13. 这个模块的工程结论

Python Agent 工程的重点是把不确定模型放进确定代码结构里：

1. 用清晰数据结构管理状态。
2. 用装饰器和注册表治理工具。
3. 用 asyncio 提升 IO 并发，但要限流。
4. 用统一 LLM client 管超时、重试、成本和 fallback。
5. RAG pipeline 分步实现，方便评估。
6. 多轮对话要摘要、检索和状态分离。
7. 工具结果和大数据要先处理再给模型。

Agent 的智能来自模型，但工程质量来自这些朴素的 Python 结构。

## 14. 外部资料补强：Python 工程要用标准库能力托住 Agent 循环

Python 官方 asyncio 文档强调，直接调用 coroutine 不会执行它，必须 await、create_task 或通过 TaskGroup 调度。这个细节在 Agent 并发工具调用里很容易踩坑：如果忘记 await，工具根本没跑；如果 fire-and-forget 没保存 task 引用，后台任务可能失控或丢失。

`asyncio.gather` 适合并发等待多个 awaitable，但 Python 文档也提示 TaskGroup 提供更强的结构化并发安全性。对 Agent 来说，如果多个并行工具中一个失败，通常希望取消或明确处理其他分支，而不是让异常悄悄传播。因此复杂并发工具执行更适合 TaskGroup 或显式错误收集。

dataclasses 官方文档提供了轻量结构化数据对象，这很适合 Agent State、Tool Result、Eval Result。相比到处传 dict，dataclass 能让字段更清晰，也方便序列化、日志和测试。复杂校验可以再升级到 Pydantic，但 dataclass 是非常好的起点。

concurrent.futures 文档说明了 ThreadPoolExecutor 和 ProcessPoolExecutor 的不同用途。Agent 中 IO 密集任务适合 asyncio 或线程，CPU 密集任务例如本地 embedding、PDF 解析、静态分析，则更适合进程池或独立服务。不要用一个事件循环硬扛所有工作。

## 15. 深度补充：Python 不是胶水，而是 Agent 的控制面

很多人把 Python 在 Agent 里的角色理解成“调 API 的胶水代码”。这会低估工程风险。Agent 的核心循环、工具注册、状态管理、RAG pipeline、并发调度、流式输出、评估回放，最后大多都落在 Python 控制面里。

模型输出是不确定的，但 Python 控制面应该尽量确定：

| Agent 不确定性 | Python 控制面要做什么 |
| --- | --- |
| 模型可能选错工具 | Tool Registry、Schema 校验、权限检查 |
| 模型可能循环太久 | max_steps、timeout、cost budget |
| 工具可能失败 | 重试、错误分类、observation |
| 状态可能污染 | dataclass/Pydantic、copy、checkpoint |
| 并发可能失控 | semaphore、TaskGroup、backpressure |
| 输出可能不合规 | parser、guardrail、结构化输出 |

所以 Python 工程不是“基础语法”，而是在给概率模型搭一个确定的运行笼架。写得松散，模型问题会被放大；写得清晰，模型能力才能稳定释放。

## 16. 应用方式：一个可维护 Agent 工程的模块划分

一个中等复杂度的 Python Agent 项目，可以按下面方式拆：

```text
agent_app/
  agents/
    planner.py
    executor.py
  tools/
    registry.py
    schemas.py
    order_tools.py
    search_tools.py
  rag/
    ingest.py
    retrieve.py
    rerank.py
  runtime/
    loop.py
    context.py
    state.py
    policy.py
    tracing.py
  evals/
    datasets/
    run_eval.py
    judges.py
  clients/
    llm.py
    vector_db.py
  tests/
```

这种划分的好处是：模型逻辑、工具实现、运行控制、评估脚本彼此独立。工具可以单测，RAG 可以单独评估，Agent loop 可以回放 trace，LLM client 可以替换模型提供商。

最不推荐的是把所有东西写在一个 `main.py` 里：先拼 prompt，再调用模型，再 if/else 执行工具，再拼下一轮 messages。Demo 可以这样，生产会很快变成无法回放、无法评估、无法限权的黑盒。

## 17. 为什么 asyncio 常常是 Agent 的第一道性能杠杆

Agent 的慢，很多时候不是模型单次调用慢，而是多个 IO 被串行执行：搜索、RAG、rerank、工具 API、多个 judge、多个子任务。如果这些步骤彼此独立，就应该并发。

但 `asyncio.gather` 不是万金油。生产里要考虑：

| 问题 | 处理 |
| --- | --- |
| 下游 rate limit | 用 semaphore 限并发 |
| 单个任务失败 | 收集异常或 TaskGroup 结构化取消 |
| 超时 | `asyncio.timeout` 或客户端 timeout |
| 用户取消 | 取消关联任务，清理资源 |
| 部分成功 | 返回可用结果和失败原因 |
| 背压 | 队列满时拒绝或延迟处理 |

并发工具执行的目标不是“同时跑更多”，而是在不打爆下游的前提下降低端到端延迟。对 LLM API、向量库、搜索 API 来说，限流和退避与并发同样重要。

## 18. 代码骨架：最小可治理 ReAct Loop

下面是一个概念骨架，不追求可直接运行，而是展示边界应该放在哪里：

```python
async def run_agent(task: str, runtime: Runtime, max_steps: int = 8) -> AgentResult:
    state = AgentState(task=task)

    for step in range(max_steps):
        messages = runtime.context.build(state)
        model_output = await runtime.llm.generate(
            messages=messages,
            tools=runtime.tools.visible_schemas(state),
            timeout=runtime.policy.model_timeout,
        )

        runtime.trace.record_model_step(step, messages, model_output)

        if model_output.final_answer:
            return runtime.output.validate(model_output.final_answer, state)

        for call in model_output.tool_calls:
            tool = runtime.tools.get(call.name)
            args = tool.schema.validate(call.arguments)
            runtime.policy.check_tool_allowed(tool, args, state)

            try:
                observation = await runtime.executor.run(tool, args, state)
            except ToolError as exc:
                observation = Observation.from_error(exc)

            runtime.trace.record_tool_call(step, call, observation)
            state = state.with_observation(call, observation)

    return AgentResult.incomplete(state=state, reason="max_steps_exceeded")
```

这段骨架体现几个原则：上下文由 Context Builder 生成；工具由 Registry 管；参数先校验再执行；权限在 Policy 里；错误变成 observation；trace 贯穿模型和工具；达到 max_steps 时返回可解释的未完成状态。

## 19. 面试追问视角：可以怎么回答

如果被问“Python Agent 最容易出什么工程 bug”，可以答：

> 状态共享和异步并发。比如多个分支修改同一个 dict、忘记 await coroutine、没有 timeout、工具异常没有转成 observation、messages 无限增长、工具结果过大撑爆上下文。这些都不是模型问题，是控制面问题。

如果被问“为什么要封装 LLM client”，可以答：

> 因为生产需要统一处理 timeout、retry、rate limit、streaming、token 统计、日志脱敏、模型路由和 fallback。直接在业务里到处调 SDK，会让治理逻辑散落，后续换模型或做回放很难。

如果被问“如何实现一个工具注册系统”，可以答：

> 用装饰器注册普通函数，同时保存 name、description、JSON schema、handler、permission、timeout、risk level。模型只看到 schema，执行前由注册表找到 handler，再经过 schema 校验和 policy check。
