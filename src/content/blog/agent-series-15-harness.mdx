Harness 是生产 Agent 里最容易被低估的一层。LLM 负责生成下一步判断，框架提供一些编排抽象，但真正把任务、上下文、工具、状态、权限、预算、沙箱、trace 和终止条件组织起来的，是 Harness。

`15_Harness` 模块可以看作整套 Agent 知识的收束：如果前面的模块分别讲模型、Prompt、RAG、工具、框架、安全和评估，那么 Harness 就是把它们放进一个可运行环境的控制层。

## 1. Harness 是什么

Agent Harness 是运行控制层。它不等同于 prompt，也不等同于框架，更不是某一个工具函数。它负责把模型放进受控任务环境中：

| 责任 | 说明 |
| --- | --- |
| 执行控制 | Agent Loop、终止条件、步数上限 |
| 上下文管理 | 决定每轮模型看到什么 |
| 工具治理 | Tool Schema、调用、异常、权限 |
| 状态管理 | 保存当前任务进度和中间结果 |
| 记忆管理 | 注入和更新长期经验 |
| 安全策略 | 审批、沙箱、权限、注入防御 |
| 可观测性 | trace、日志、指标、成本统计 |

没有 Harness，模型只能回答文本。有了 Harness，模型才能观察环境、行动、接收反馈、继续推理，并在边界内停止。

## 2. Harness 与 LLM、Framework、Runtime 的关系

| 概念 | 作用 |
| --- | --- |
| LLM | 生成推理、动作和答案 |
| Agent Framework | 提供编排和组件抽象 |
| Agent Runtime | 实际执行任务的运行环境 |
| Harness | 控制每轮上下文、工具、状态、预算、安全和终止 |

框架可以提供 Harness 能力，但 Harness 是更抽象的运行机制。一个团队可以用 LangGraph 实现 Harness，也可以自研状态机实现 Harness。

## 3. 为什么能力不只取决于模型

同一个模型，在不同 Harness 下表现会差很多。原因是 Harness 决定了：

1. 模型每轮看到哪些上下文。
2. 可用工具是否清晰可靠。
3. 工具失败后如何反馈。
4. 是否有记忆和检索。
5. 是否有预算和终止边界。
6. 是否有 verifier 纠偏。

弱 Harness 会让强模型迷路；强 Harness 能让中等模型稳定完成窄任务。

## 4. 基础 Harness 组件

一个基础 Harness 至少包含：

| 组件 | 功能 |
| --- | --- |
| Task Parser | 解析用户目标 |
| Context Builder | 拼系统指令、状态、记忆、工具 |
| Model Client | 调用 LLM |
| Tool Registry | 管理工具 |
| Executor | 执行工具调用 |
| State Store | 保存任务状态 |
| Policy Engine | 权限和预算控制 |
| Trace Recorder | 记录完整轨迹 |

这些组件不一定复杂，但边界要清晰。尤其是 Context Builder 和 State Store，决定了 Agent 是否能连续执行。

## 5. Agent Loop

Harness 控制 Agent Loop：

1. 构建上下文。
2. 调用模型。
3. 解析动作。
4. 校验权限和参数。
5. 执行工具。
6. 写入 observation。
7. 更新状态和记忆。
8. 判断是否完成或停止。

循环必须有硬限制：step limit、token limit、cost limit、time limit。不要只靠模型自觉结束。

## 6. 每轮该看到什么上下文

Context Builder 是 Harness 的核心。它要在 token 预算内选择最有用信息：

| 内容 | 处理方式 |
| --- | --- |
| 系统规则 | 必须注入，但保持精简 |
| 用户目标 | 当前任务核心 |
| 任务状态 | 计划、已完成步骤、变量 |
| 最近轨迹 | 保留关键 observation |
| 长期记忆 | 按相关性召回 |
| RAG 证据 | 按需注入，带来源 |
| 工具 Schema | 只暴露本轮相关工具 |
| 安全策略 | 高风险约束和拒绝规则 |

好的 Harness 不是把所有信息都给模型，而是每轮给刚好足够的信息。

## 7. 工具管理

Harness 负责工具生命周期：注册、选择暴露、参数校验、权限检查、执行、重试、结果截断、错误转换。

工具失败后，Harness 应把可读错误作为 observation 返回模型，同时记录结构化 trace。对高风险工具，Harness 要在执行前触发审批，而不是让模型自己判断要不要审批。

## 8. Tool Schema 对稳定性的影响

工具 Schema 是 Harness 和模型之间的动作合同。Schema 模糊会直接导致工具误用。

Harness 可以增强稳定性：

1. 根据任务动态选择工具子集。
2. 对参数做 JSON Schema 校验。
3. 对业务权限做二次校验。
4. 对返回结果做结构化包装。
5. 对错误做可修复描述。

## 9. 状态管理与记忆管理

Harness 要区分状态和记忆：

| 类型 | 内容 | 典型存储 |
| --- | --- | --- |
| State | 当前任务进度、计划、变量、工具结果 | Redis、DB、checkpoint |
| Short Context | 当前会话最近消息 | messages |
| Long-term Memory | 用户偏好、经验、跨任务事实 | 向量库、数据库 |

状态是任务执行的真相，记忆是可召回的经验。状态应该确定、可恢复；记忆可以语义检索，但要防污染。

## 10. 预算限制

Harness 中常见限制：

| 限制 | 解决问题 |
| --- | --- |
| step limit | 防止无限循环 |
| token limit | 控制上下文和模型成本 |
| cost limit | 防止异常烧钱 |
| time limit | 保护用户体验和资源 |
| tool call limit | 防止工具滥用 |
| memory limit | 防止上下文膨胀 |

这些限制要作为系统策略执行。达到限制后，Agent 应输出当前状态、已完成内容和未完成原因，而不是沉默失败。

## 11. 权限、审批与沙箱

Harness 是安全策略落地的位置。它知道当前用户是谁、任务风险是什么、工具权限如何、是否需要人工审批。

高风险操作包括：删除数据、发送外部消息、转账、修改权限、执行代码、访问敏感数据。Harness 应在这些操作前暂停，展示计划和参数，等待确认。

代码执行必须进入沙箱，并控制网络、文件系统、CPU、内存和时间。

## 12. 单 Agent Harness 与多 Agent Harness

单 Agent Harness 控制一个循环。多 Agent Harness 还要解决：

| 问题 | 说明 |
| --- | --- |
| 任务分配 | 哪个 Agent 负责什么 |
| 上下文隔离 | 每个 Agent 看哪些信息 |
| 共享状态 | 子结果如何汇总 |
| 通信协议 | Agent 之间如何传消息 |
| 冲突解决 | 结果不一致谁裁决 |
| 全局预算 | 多个 Agent 共享成本上限 |
| Trace 合并 | 能回放完整协作路径 |

多 Agent Harness 的复杂度远高于单 Agent。要先证明分工收益，再引入。

## 13. 为什么调试需要 trace

Agent 失败很少能靠最终答案定位。Trace 要记录每轮：

1. 输入上下文摘要。
2. 模型输出。
3. 工具调用名和参数。
4. 工具结果。
5. 状态变化。
6. token、成本、耗时。
7. 策略判断和审批。

Trace 是 Agent 的调用栈。没有 trace，就没有真正的生产调试。

## 14. Evaluation Harness 与 Runtime Harness

Runtime Harness 服务线上任务，关注安全、延迟、成本和可靠性。Evaluation Harness 服务评估任务，关注可重复、可比较、可统计。

| 维度 | Runtime Harness | Evaluation Harness |
| --- | --- | --- |
| 输入 | 用户真实请求 | 固定测试集 |
| 目标 | 完成任务 | 测量质量 |
| 输出 | 用户结果 | 指标和失败样本 |
| 控制 | 权限、预算、降级 | 固定种子、版本、评分 |
| 关注 | SLA 和安全 | 回归和优化 |

成熟团队会同时建设两套 Harness，并让线上失败回流到 Evaluation Harness。

## 15. 这个模块的工程结论

Harness 是 Agent 生产化的核心：

1. LLM 负责决策，Harness 负责运行边界。
2. 每轮上下文都应由 Harness 精准选择。
3. 工具调用必须经过 Schema、权限和策略校验。
4. 状态、记忆和上下文要分开管理。
5. step、token、cost、time limit 缺一不可。
6. 高风险动作需要审批，代码执行需要沙箱。
7. Trace 是调试和评估的基础。
8. Runtime Harness 与 Evaluation Harness 要形成闭环。

如果说模型是 Agent 的大脑，Harness 就是神经系统、骨架和刹车。没有 Harness，Agent 只能展示能力；有了 Harness，Agent 才能进入生产。

## 16. 外部资料补强：Harness 正在被 SDK 和运行时显式化

OpenAI Agents SDK 文档把 Agent、Runner、tools、guardrails、handoffs、sessions 和 tracing 组织在一起，这几乎就是 Harness 的产品化表达。文档也指出，如果想让 SDK 管理 turns、tools、guardrails、handoffs 和 sessions，就用 Agent 加 Runner；如果想自己拥有循环，则直接使用底层 API。这正好说明 Harness 的本质是“谁控制 Agent loop”。

Agents SDK 的 tracing 文档强调，trace 会记录 LLM generations、tool calls、handoffs、guardrails 和自定义事件。这个设计与本篇“trace 是 Agent 调用栈”的观点一致。没有 trace，Agent 的失败无法被系统性复盘。

Agents SDK 的 guardrails 和 human review 文档则把输入、输出、工具输入、工具输出的验证显式放进运行时。对 Harness 来说，这意味着安全不应只写在 Prompt 里，而应成为每次 run 的结构化检查。

MCP 文档把 Host、Client、Server、Tools、Resources、Prompts 区分开。这个模型有助于 Harness 管理外部能力边界：有的 MCP server 只提供资源，有的暴露强动作工具；Harness 必须根据风险选择暴露什么、何时暴露、如何审批。

Anthropic 的 Agent 构建文章也强调 stopping conditions、sandboxed testing、ground truth from environment 和 human checkpoints。换句话说，生产 Agent 的关键不是让模型持续运行，而是让它在环境反馈、停止条件和人类检查点之间运行。

## 17. 深度补充：Harness 解决的是“谁来控制循环”

Agent 系统最关键的问题不是“模型会不会推理”，而是“谁来控制循环”。如果模型自己决定看什么、用什么工具、何时停止、是否需要审批、错误怎么恢复，那么系统就把运行控制权交给了概率模型。Harness 的出现，是为了把这些控制权收回到工程系统。

OpenAI Agents SDK 的 Runner loop 很典型：模型产生 final output、handoff 或 tool call；运行时执行工具、追加结果、继续循环；超过 max turns 就停止。这个结构说明一个事实：Agent 的能力来自模型和工具，但稳定性来自运行时控制。

所以 Harness 不是一个可有可无的封装层。它回答的是生产 Agent 的核心控制问题：每一轮模型看到什么，能做什么，做错怎么办，什么时候停，如何复盘。

## 18. 应用方式：Harness 如何托住不同类型 Agent

不同 Agent 的 Harness 重点不同。

| Agent 类型 | Harness 重点 |
| --- | --- |
| 客服 Agent | 身份注入、订单工具权限、人工转接、政策引用 |
| 代码 Agent | 文件变更边界、测试循环、命令沙箱、diff 审计 |
| 数据分析 Agent | 数据源权限、脚本执行、图表产物、可复现 notebook |
| 研究 Agent | 搜索来源、引用核验、claim verification、报告版本 |
| 运维 Agent | 告警上下文、只读诊断优先、变更审批、回滚 |
| 多 Agent 系统 | 子任务分配、共享状态、预算隔离、trace 合并 |

这说明 Harness 不是通用模板一套到底，而是一组运行控制能力。不同业务可以共用基础组件：context builder、tool manager、policy engine、trace recorder、state store；但风险策略、工具集和终止条件要按场景定制。

## 19. Runtime Harness 与 Evaluation Harness 如何闭环

Runtime Harness 负责线上执行，Evaluation Harness 负责离线评估。它们应该共享同一套 trace 和版本信息。

闭环可以这样设计：

1. Runtime Harness 记录每次任务的 prompt version、model version、tool schema version、trace、成本和结果。
2. 线上失败或人工差评被采样出来。
3. Evaluation Harness 用相同输入和固定版本回放。
4. 人工或 LLM judge 标注根因。
5. 修改 prompt、tool、RAG 或 workflow。
6. 跑回归集，确认没有退化。
7. 小流量灰度，再观察 Runtime 指标。

这就是为什么 Harness 和 Eval 不能分家。没有 Runtime trace，Eval 不知道该测什么；没有 Eval 回归，Runtime 只能在线上试错。

## 20. 为什么 SDK 不是 Harness 的全部

OpenAI Agents SDK、LangGraph、MCP、CrewAI 这类工具都能提供 Harness 的一部分能力，但它们不等于完整生产 Harness。

完整生产 Harness 还需要和企业系统结合：

| 能力 | 框架/SDK 常提供 | 企业还要补 |
| --- | --- | --- |
| Agent loop | Runner、Graph、tool call | 任务 SLA、取消、恢复策略 |
| Tool 管理 | Schema、调用接口 | 业务权限、审批、审计、幂等 |
| Guardrail | 输入/输出检查 | 合规策略、红队样本、事件响应 |
| Trace | Span 和事件 | 数据脱敏、租户隔离、留存策略 |
| State | Session/checkpoint | 多租户存储、灾备、权限访问 |
| Handoff | Agent 切换 | 组织流程和人工责任归属 |

这也是工程取舍：用 SDK 可以加速搭建，但核心边界要由业务自己定义。否则系统很快会遇到“框架能跑，但企业不能上线”的问题。

## 21. Harness 的反模式

| 反模式 | 结果 | 修正方式 |
| --- | --- | --- |
| 把 loop 写在 prompt 里 | 停止条件不可靠 | 代码层 max_steps/time/cost |
| 每轮塞全部上下文 | 成本高、注意力稀释 | Context Builder 动态选择 |
| 工具错误直接抛出 | Agent 无法恢复 | 转成结构化 observation |
| 高风险动作自动执行 | 安全事故 | Policy + human approval |
| 没有 trace | 失败不可复盘 | 全链路 span 和状态快照 |
| Runtime 和 Eval 分离 | 线上问题无法回归 | trace 回流评估集 |
| 多 Agent 共享一坨上下文 | 信息污染和责任不清 | 上下文隔离、共享状态、协议输出 |

## 22. 面试追问视角：可以怎么回答

如果被问“为什么有框架还要 Harness”，可以答：

> 框架提供组件和编排，但 Harness 是生产运行控制层。它决定上下文、工具、状态、权限、预算、终止、trace、审批和恢复。框架可以实现其中一部分，但业务风险和运行边界需要自己设计。

如果被问“Agent loop 如何停止”，可以答：

> 不能只靠模型说完成。要有 max_steps、time limit、token/cost budget、任务状态判定、verifier、用户确认和错误阈值。停止时要返回已完成内容、未完成原因和下一步建议。

如果被问“如何调试一个失败 Agent”，可以答：

> 看 trace：每轮输入上下文、模型输出、工具调用、参数、工具结果、状态变化、策略判断、token 和耗时。然后把失败归因到 prompt、tool、RAG、model、workflow、policy 或外部依赖，再加入 eval 回归。
